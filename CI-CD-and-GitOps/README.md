## Advanced Version Control

### 1. Модели ветвления: Git Flow vs Trunk-Based Development

Выбор стратегии ветвления определяет скорость доставки фич, частоту релизов и подход к автоматизации CI/CD пайплайнов.



#### Git Flow
Классическая, строго регламентированная модель ветвления, завязанная на стабильные релизы.
* **Основные ветки:** `main/master` (продуктовая стабильная база) и `develop` (основная ветка интеграции фич).
* **Вспомогательные ветки:** `feature/*` (для разработки фич), `release/*` (подготовка релиза, стабилизация, багфиксы перед продакшеном), `hotfix/*` (срочные патчи прод-уязвимостей в обход develop).
* **Плюсы:** Идеальный контроль над качеством кода, жесткий аудит (Code Review) на каждом этапе, отличный выбор для enterprise-продуктов с редкими, плановыми релизами (например, раз в месяц) или коробочного ПО.
* **Минусы:** Медленный Time-to-Market. Огромная боль с накоплением технического долга и регулярными **«Merge Hells»** (адскими конфликтами при слиянии веток, которые жили отдельно неделями). Практически несовместим с концепцией истинного Continuous Delivery.



#### Trunk-Based Development (TBD)
Современный облачный стандарт разработки, ориентированный на высокую скорость (High-Velocity Engineering).
* **Концепция:** Все разработчики коммитят в одну основную ветку (`main/trunk`) напрямую или через очень короткоживущие (Short-lived) ветки (живут не дольше 1-2 дней).
* **Плюсы:** Минимальный риск конфликтов, фичи моментально интегрируются в общую кодовую базу. Максимально быстрая обратная связь от CI-пайплайнов. Фундамент для Continuous Deployment.
* **Минусы:** Требует высочайшей инженерной культуры, 100% покрытия кода автотестами и зрелого пайплайна. Если в `main` улетит сломанный код — заблокируется работа всей команды.
* **Бизнес-логика на TBD (Feature Flags):** Чтобы фича, которая еще не готова полностью, улетела в прод и ничего не сломала, её прячут под **Feature Toggles / Feature Flags** (динамические переключатели в коде). Фича деплоится на прод в «выключенном» состоянии, а затем активируется через админку без перезапуска сервисов.

### 2. Слияние веток: Merge vs Rebase

Главный холивар на собеседованиях. Оба инструмента решают задачу интеграции изменений из одной ветки в другую, но делают это принципиально по-разному.

| Критерий | Git Merge | Git Rebase |
| :--- | :--- | :--- |
| **Принцип работы** | Создает специальный **Merge Commit** (коммит слияния), у которого два родителя. Объединяет две истории «как есть». | Берутся коммиты из твоей ветки, временно откладываются, твоя ветка обновляется до актуального состояния целевой ветки, а затем твои коммиты **переприменяются** сверху (Replay). |
| **История коммитов** | **Недеструктивная.** Полностью сохраняет историческую хронологию разработки (видно, когда ветка началась, куда ходила и где слилась). История нелинейная (граф выглядит как "паутина"). | **Деструктивная (Перезапись истории).** Переписывает хэши коммитов, создавая их заново. Делает историю коммитов идеально **линейной** и чистой (как будто разработки в боковых ветках никогда не было). |
| **Работа с конфликтами** | Все конфликты решаются один раз — непосредственно в момент создания Merge-коммита. | Конфликты могут возникать последовательно **на каждом переприменяемом коммите**, что требует пошагового разрешения. |

#### Главное золотое правило Rebase (The Golden Rule)
> **Никогда не делай rebase для веток, которые уже улетели в удаленный репозиторий (Push) и над которыми работают другие люди (например, main или develop).** Переписав историю публичной ветки, ты сломаешь локальные репозитории всех своих коллег, так как хэши коммитов изменятся, и Git сойдет с ума при попытке синхронизации.

*Типичный DevOps-workflow:* Пишешь фичу локально $\rightarrow$ подтягиваешь изменения из мастера через `git pull --rebase origin main` (держишь свою ветку линейной и актуальной) $\rightarrow$ перед вливанием в прод делаешь `Squash` (схлопываешь 20 мелких грязных коммитов вроде "fix typo" в один красивый "feat: add news worker") $\rightarrow$ вливаешь в главную ветку.

### 3. Разрешение конфликтов (Merge Conflicts Resolution)

Конфликт возникает, когда два разработчика одновременно изменили одни и те же строки в одном и том же файле в разных ветках, и Git не может самостоятельно решить, какую версию оставить.

#### Алгоритм траблшутинга и разрешения конфликта:

1. **Локализация конфликта:** При попытке сделать `merge` или `rebase` Git останавливает процесс и выводит список проблемных файлов. Внутри файлов появляются маркеры конфликта:
   ```text
   <<<<<<< HEAD
   const server = fastify({ logger: true }); // Твои локальные изменения
   =======
   const server = fastify({ logger: false }); // Изменения из ветки, которую ты вливаешь
   >>>>>>> feature/disable-logs
   ```

2. **Выбор стратегии разрешения:**
   * **Ручной разбор:** Инженер открывает файлы в IDE (VS Code / WebStorm) или через CLI-утилиты (`git mergetool`) и решает, какую строку оставить, удаляя маркеры `<<<<<<<`, `=======` и `>>>>>>>`.
   * **Автоматический выбор стороны (если точно уверен):**
     * Выбрать только свои изменения: `git checkout --ours path/to/file.ts`
     * Выбрать только входящие изменения: `git checkout --theirs path/to/file.ts`

3. **Завершение процесса:**
   * После ручной правки файлы обязательно добавляются в индекс: `git add path/to/file.ts`
   * Если это был merge: завершаем операцию командой `git merge --continue` (или просто `git commit`).
   * Если это был rebase: продолжаем процесс командой `git rebase --continue`.
   * Если всё пошло не так и ты запутался, всегда можно безопасно откатиться к исходному состоянию: `git merge --abort` или `git rebase --abort`.

## CI/CD Pipelines Architecture

### 1. Этапы пайплайна (Pipeline Stages) и их логика

Правильно спроектированный пайплайн должен обеспечивать быстрый фидбэк разработчику (Fast Feedback Loop) и гарантировать стабильность сборки. Этапы всегда выполняются от быстрых и дешевых к долгим и дорогим.



1. **Build (Сборка):** * Компиляция исходного кода (для компилируемых языков) или сборка фронтенда (Webpack/Vite).
   * Линтинг и статический анализ (`eslint`, `prettier`), проверка синтаксиса.
   * Установка зависимостей (например, `npm ci` или `pnpm install`).
2. **Test (Тестирование):**
   * **Unit Tests (Модульные тесты):** Изолированная проверка логики функций (выполняется очень быстро, исчисляется секундами).
   * **Integration Tests (Интеграционные тесты):** Проверка взаимодействия компонентов (например, бэкенда с реальной или поднятой в Docker базой данных PostgreSQL через Prisma).
3. **Security Scan (Анализ безопасности / DevSecOps):**
   * **SAST (Static Application Security Testing):** Анализ исходного кода на уязвимости (например, *SonarQube*, *Trivy*).
   * **SCA (Software Composition Analysis):** Проверка зависимостей и пакетов из `package.json` на наличие известных уязвимостей и CVE (например, `npm audit`, *Snyk*).
   * **Container Scanning:** Если на этом этапе собирается Docker-образ, его сканируют утилитой вроде *Trivy* или *Clair* до отправки в Registry.
4. **Deploy (Доставка):**
   * Пуш готового Docker-образа в Registry (AWS ECR, Docker Hub).
   * Обновление манифестов инфраструктуры (GitOps) или прямой деплой на сервера/кластер.

### 2. Управление артефактами и стратегии кэширования

Главная проблема неэффективных пайплайнов — они тратят кучу времени на повторение одних и тех же действий (скачивание пакетов, сборка статики). Для ускорения используются два инструмента, которые важно четко различать.

| Критерий | Artifacts (Артефакты) | Caching (Кэширование) |
| :--- | :--- | :--- |
| **Основная цель** | Передача результатов работы **между стадиями (Jobs)** внутри *одного и того же* запуска пайплайна. | Ускорение сборки за счет переиспользования данных **между разными** (предыдущими и текущими) запусками пайплайна. |
| **Пример данных** | Скомпилированный бинарник, папка `dist/` с собранным фронтендом, сгенерированные файлы отчетов. | Папка `node_modules/`, кэш пакетного менеджера (`~/.npm` или `~/.pnpm-store`), кэш слоев Docker-образов. |
| **Жизненный цикл** | Создаются на этапе `Build`, живут до конца текущего пайплайна и удаляются (или хранятся заданное время). | Сохраняются в конце успешного пайплайна и скачиваются в начале следующего, если совпал ключ кэша (Cache Key). |

#### Идеальный паттерн настройки кэша для Node.js / TypeScript:
Кэш не должен инвалидироваться на каждую сборку, если зависимости не менялись. Для этого ключ кэша привязывают к хэш-сумме файла зависимостей.
* *Ключ кэша:* `cache-node-modules-{{ checksum "package-lock.json" }}` (или `pnpm-lock.yaml`).
* *Логика:* Если разработчик поправил только код приложения, хэш `package-lock.json` останется прежним, пайплайн пропустит стадию скачивания пакетов из интернета и мгновенно развернет `node_modules` из кэша. Если добавился новый пакет — хэш изменится, кэш сбросится, и соберется новый.

### 3. Безопасность в CI/CD: Secrets Management & OIDC

Пайплайн обладает огромными привилегиями: он имеет доступ к продакшен-серверам и базам данных. Взлом CI/CD — кратчайший путь для злоумышленника в инфраструктуру компании.

#### Менеджмент секретов (Secrets Management)
* **Антипаттерн:** Хардкодить пароли, токены и приватные ключи в код репозитория или конфигурационные файлы.
* **Базовый уровень:** Использование защищенных переменных окружения самого CI-сервиса (GitLab CI/CD Variables, GitHub Actions Secrets). Они маскируются в логах (вместо пароля выводится `***`), но всё равно хранятся в базе данных провайдера.
* **Enterprise уровень:** Пайплайн при запуске динамически запрашивает секреты из специализированных внешних хранилищ (например, *HashiCorp Vault*, *AWS Secrets Manager*), используя краткосрочные токены. Секрет живет только в оперативной памяти воркера во время выполнения джобы.

#### Концепция OIDC (OpenID Connect): Пайплайны без постоянных ключей

Исторически, чтобы задеплоить что-то в AWS из GitHub Actions, инженеры создавали IAM-пользователя, генерировали для него постоянные `AWS_ACCESS_KEY_ID` и `AWS_SECRET_ACCESS_KEY` и прописывали их в секреты репозитория. 
* *Риск:* Если репозиторий взломают или секреты случайно утекут, злоумышленник получит вечный доступ к облаку.

**Современный паттерн с OIDC (Беспарольный доступ):**
Вместо использования постоянных ключей, твой CI-провайдер и облачный провайдер настраивают между собой федеративное доверие.

1. Пайплайн запускается на воркере (например, в GitHub Actions).
2. GitHub генерирует временный, подписанный криптографический токен — **OIDC ID Token** (JWT), внутри которого написано: *«Я пайплайн репозитория daniil/agentic-news, запускаюсь из ветки main»*.
3. Пайплайн стучится в облако (AWS STS) и говорит: *«Я хочу принять роль `DeployerRole`. Вот мой JWT-токен от GitHub»*.
4. AWS проверяет подпись GitHub (через открытые ключи), парсит JWT и смотрит в настройки своей IAM Роли: *«Так, этой роли разрешено принимать запросы от репозитория daniil/agentic-news. Всё сходится»*.
5. AWS STS выдает пайплайну **временные access-ключи**, которые автоматически превратятся в тыкву через 15-60 минут. Никаких постоянных секретов в репозитории не хранится вообще.

## Deployment Strategies (Стратегии деплоя)

Выбор стратегии развертывания определяет, как новые версии твоего приложения будут доставляться до пользователей и как система будет реагировать на возможные сбои. Главная цель — минимизировать или полностью исключить время простоя (Downtime) и снизить риски.

---

### 1. Recreate (Пересоздание)

Самый простой и радикальный метод деплоя. Старая версия приложения полностью уничтожается до того, как начнется подъем новой версии.



* **Как это работает:** 1. Трафик идет на приложения Версии 1.
  2. Все инстансы Версии 1 одновременно выключаются (Downtime начался).
  3. Запускаются инстансы Версии 2, проходят проверки (Health Checks).
  4. Трафик перенаправляется на Версию 2 (Downtime закончился).
* **Плюсы:** * Абсолютно исключена ситуация, когда в системе одновременно запущены две разные версии кода (нет проблем с обратной совместимостью схем БД или API).
  * Не требует избыточных ресурсов инфраструктуры (CPU/RAM).
* **Минусы:** * Наличие гарантированного **даунтайма** на время, пока старые контейнеры тушатся, а новые скачиваются и стартуют.
  * Долгое и болезненное время отката (Rollback): если Версия 2 упала при старте, нужно заново запускать старый пайплайн.
* **DevOps-применение:** Идеально подходит для окружений разработки (Dev/Stage), внутренних аналитических систем или тяжелых Stateful-сервисов, которые физически не могут работать параллельно в нескольких экземплярах.

---

### 2. Rolling Update (Постепенное обновление)

Дефолтная стратегия в Kubernetes. Инстансы старой версии заменяются новыми постепенно, по очереди или небольшими группами (батчами).



* **Как это работает:** 1. Балансировщик распределяет трафик между четырьмя подами Версии 1.
  2. Запускается первый под Версии 2. Как только он успешно проходит Health Check, балансировщик включает его в ротацию.
  3. Один старый под Версии 1 уничтожается.
  4. Процесс повторяется для остальных подов по цепочке, пока все 100% инстансов не обновятся.
* **Параметры тонкой настройки (в Kubernetes):**
  * `maxSurge`: Сколько дополнительных подов сверх целевого количества можно поднять в процессе деплоя (например, `25%` разрешит K8s временно поднять еще один под, увеличив их число до 5).
  * `maxUnavailable`: Сколько подов от целевого количества могут быть недоступны в процессе обновления (например, `25%` разрешит сразу прибить один старый под, не дожидаясь подъема нового).
* **Плюсы:** * **Zero Downtime** — пользователи вообще не замечают процесса обновления.
  * Нагрузка на инфраструктуру возрастает незначительно (только на величину `maxSurge`).
* **Минусы:** * В процессе деплоя в кластере одновременно живут Версия 1 и Версия 2. Твой бэкенд обязан поддерживать обратную совместимость: если Версия 2 уже накатила миграцию в БД, Версия 1 не должна от этого сломаться.
  * Сложно контролировать откат, если баг обнаружился в середине процесса.

---

### 3. Blue-Green Deployment

Стратегия, при которой одновременно развернуты две абсолютно идентичные и полноценные среды (прод-окружения). Одна из них активно принимает трафик («Синяя»), а вторая стоит наготове с новой версией кода («Зеленая»).



* **Как это работает:**
  1. `Blue` (Синяя) — текущий прод, туда идет 100% пользовательского трафика.
  2. DevOps-пайплайн разворачивает новую версию приложения в изолированную среду `Green` (Зеленая).
  3. Внутри `Green` проводятся полноценные смоук-тесты, QA-инженеры проверяют работоспособность.
  4. Как только всё подтверждено, на уровне маршрутизатора (DNS или Load Balancer) происходит мгновенное переключение: 100% трафика перенаправляется на `Green`.
  5. Среда `Blue` остается «замороженной» на несколько часов/дней. Если на проде вылезет критический баг, переключение обратно на старую рабочую версию займет ровно одну секунду. После успешного релиза `Blue` гасится или становится плацдармом для следующего деплоя.
* **Плюсы:**
  * Нулевой даунтайм и **моментальный откат (Instant Rollback)** в случае аварии.
  * Тестирование происходит на реальном прод-железе до того, как туда пустят пользователей.
* **Минусы:**
  * **Double Costing** (Двойная цена). На время деплоя тебе нужно держать в два раза больше серверов, ядер CPU и гигабайт RAM.

---

### 4. Canary Deployment (Канареечный деплой)

Название пошло от шахтеров, которые брали с собой в забой канареек: если птица гибла от метана, люди успевали эвакуироваться. В IT эта стратегия означает раскатку новой версии кода на **минимальный процент реальных пользователей**.



* **Как это работает:**
  1. У тебя запущено 100 подов с Версией 1.
  2. Пайплайн разворачивает рядом всего 2-5 подов с Версией 2 (это «канарейки»).
  3. Балансировщик настраивается так, чтобы пускать туда фиксированный процент трафика (например, 5% случайных юзеров или пользователей из конкретного региона/с определенными заголовками).
  4. Система мониторинга (Prometheus/CloudWatch) начинает пристально следить за «канарейками»: оценивается уровень ошибок (HTTP 5xx), задержки ответов (Latency) и утилизация ресурсов.
  5. Если метрики в норме — процент трафика плавно повышают: 10% $\rightarrow$ 25% $\rightarrow$ 50% $\rightarrow$ 100%. Если пошли ошибки — деплой останавливается, а трафик на «канарейки» мгновенно перекрывается.
* **Плюсы:**
  * Минимальный радиус поражения (Blast Radius). Если новая версия содержит баг, от него пострадает лишь ничтожная часть пользователей.
  * Тестирование стабильности кода происходит на живом, реальном продакшен-трафике.
* **Минусы:**
  * Самая сложная в реализации и автоматизации стратегия. Требует продвинутых инструментов маршрутизации (Service Mesh вроде *Istio*, умных Ingress-контроллеров) и настроенных автоматических триггеров отката на основе метрик.

---

### Сводная шпаргалка по стратегиям деплоя

| Стратегия | Наличие Downtime | Риск для прода | Затраты на ресурсы | Сложность настройки |
| :--- | :--- | :--- | :--- | :--- |
| **Recreate** | Да | Средний | Низкие ($1\times$) | Низкая |
| **Rolling Update**| Нет | Средний | Низкие ($1\times + \text{maxSurge}$) | Низкая (из коробки в K8s) |
| **Blue-Green** | Нет | Низкий | Высокие ($2\times$) | Средняя |
| **Canary** | Нет | Минимальный | Низкие ($1\times$) | Высокая |

## GitOps & Declarative CD

### 1. Ядро GitOps: Ключевые принципы

**GitOps** — это парадигма и операционная модель управления инфраструктурой и девелопментом, которая берет лучшие практики разработки (версионирование, CI/CD, Code Review) и применяет их к оперированию инфраструктурой. 

Согласно официальному OpenGitOps стандарту, модель базируется на четырех фундаментальных принципах:

1. **Декларативное описание (Declarative):** Вся целевая конфигурация системы (инфраструктура, сеть, поды, права доступа) должна быть описана декларативно. Ты описываешь *желаемое состояние* (например: «хочу 3 реплики пода с образом v2»), а не шаги для его достижения. В мире Kubernetes это манифесты YAML, Helm-чарты или Kustomize-шаблоны.
2. **Версионируемое хранилище (Versioned and Immutable):** Git является единственным источником правды (**Single Source of Truth**). Желаемое состояние системы надежно зафиксировано в истории коммитов. Это гарантирует прозрачный аудит изменений и позволяет восстановить всю инфраструктуру с нуля в случае катастрофы.
3. **Автоматическое вытягивание (Approved Changes Automatically Applied):** Как только изменения в декларативном коде прошли Code Review и влились в основную ветку Git, они должны быть автоматически доставлены и применены к целевой среде.
4. **Непрерывная сверка и устранение дрейфа (Continuous Reconciliation):** Специальный программный агент в режиме реального времени сравнивает желаемое состояние из Git с фактическим состоянием в кластере. Если кто-то руками поменял конфигурацию прямо на сервере, агент фиксирует расхождение (**Configuration Drift**) и автоматически устраняет его.

---

### 2. Сравнение моделей доставки: Push vs Pull

Главная революция GitOps — это переход от классической императивной Push-модели к декларативной Pull-модели.



| Критерий | Push-модель (Классический CD) | Pull-модель (GitOps CD) |
| :--- | :--- | :--- |
| **Инструменты** | GitLab CI/CD, GitHub Actions, Jenkins. | ArgoCD, FluxCD. |
| **Где живет логика** | **Вне кластера.** Пайплайн запускается на внешнем воркере и выполняет команды вроде `kubectl apply -f manifests/`. | **Внутри кластера.** Специальный оператор запущен непосредственно в самом Kubernetes и тянет данные к себе. |
| **Доступы и безопасность** | **Небезопасно.** Внешний CI/CD сервис обязан хранить админский `kubeconfig` или токены доступа к API-серверу Kubernetes, чтобы выполнять деплой. | **Максимально безопасно.** Кластер полностью закрыт наружу. Никаких доступов внешним системам не выдается. Оператор сам забирает манифесты из Git по HTTPS/SSH. |
| **Борьба с дрейфом конфигурации** | **Отсутствует.** Пайплайн отрабатывает один раз в момент коммита. Если через час админ руками удалит под через CLI, система останется сломанной до следующего деплоя. | **Непрерывная.** Оператор крутится в бесконечном цикле (Reconciliation Loop) и мгновенно перезаписывает любые ручные изменения в кластере, возвращая систему к состоянию из Git. |

---

### 3. Архитектура ArgoCD

**ArgoCD** — это декларативный GitOps-инструмент непрерывной доставки (CD) для Kubernetes. Он реализован в виде набора кастомных контроллеров (CRD) внутри кластера.



#### Основные компоненты:
* **API Server:** Отвечает за Web UI, взаимодействие с CLI, авторизацию пользователей (RBAC, интеграция с SSO/Dex) и получение вебхуков от Git-репозиториев.
* **Repository Server (argocd-repo-server):** Внутренний сервис, который поддерживает локальный кэш Git-репозиториев. Он забирает манифесты из Git, парсит их и генерирует чистый Kubernetes YAML (рендерит Helm-чарты или Kustomize).
* **Application Controller:** Сердце ArgoCD. Он непрерывно мониторит запущенные приложения и сравнивает их живое состояние (Live State) со сгенерированным эталоном от Repository Server (Target State).

#### Ключевые состояния приложения в ArgoCD:
* `Synced` — Живое состояние в кластере на 100% соответствует коду в Git.
* `OutOfSync` — Обнаружено расхождение. Либо в Git прилетел новый коммит, который еще не применился, либо в самом кластере кто-то вручную изменил параметры ресурсов.
* `Healthy / Degraded` — Статус работоспособности самих подов (проходят ли они Liveness/Readiness пробы).

---

### 4. Стратегии синхронизации (Sync Policies)

ArgoCD позволяет гибко настроить, как именно изменения из репозитория должны применяться к кластеру.

#### Manual Sync (Ручная синхронизация)
ArgoCD подсветит статус `OutOfSync`, покажет детальный `diff` (какие конкретно строки в манифестах отличаются), но не будет ничего предпринимать. Инженер должен зайти в интерфейс или через CLI нажать кнопку **Sync**, чтобы изменения применились.
* *DevOps-применение:* Идеально для продакшен-окружений на этапе внедрения GitOps, когда команда хочет полностью контролировать момент релиза.

#### Automated Sync (Автоматическая синхронизация)
ArgoCD мгновенно применяет изменения, как только видит новый коммит в ветке. Для контроля автоматики используются две критически важные опции:

* **Prune Resources (Удаление сирот):** По умолчанию, если ты удалишь файл манифеста (например, `deployment.yaml`) из Git, ArgoCD не станет удалять этот ресурс из Kubernetes из соображений безопасности. Включение опции `Prune` разрешает ArgoCD физически уничтожать в кластере всё, чего больше нет в Git. Это гарантирует абсолютную чистоту среды.
* **Self Heal (Самолечение):** Эта опция отвечает за реакцию на ручные изменения в кластере. Если она включена, и кто-то выполнит команду `kubectl edit` или изменит количество реплик руками, ArgoCD за доли секунды зафиксирует интервенцию и автоматически накатит конфигурацию из Git поверх ручных правок.

#### Описание сущности Application в YAML-формате:
```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: agentic-news-backend
  namespace: argocd
spec:
  project: default
  source:
    repoURL: 'https://github.com/tsuyakashi/agentic-news.git' # Откуда брать код
    targetRevision: HEAD
    path: infra/manifests                             # Путь к манифестам в репозитории
  destination:
    server: 'https://kubernetes.default.svc'          # В какой кластер деплоить
    namespace: production                             # Целевой нэймспейс
  syncPolicy:
    automated:
      prune: true                                     # Удалять ресурсы, если они удалены из Git
      selfHeal: true                                  # Защита от ручных правок в кластере
```

### 5. Реактивный деплой: Оптимизация через Webhooks

По умолчанию ArgoCD работает по модели периодического опроса (**Polling**). Раз в 3 минуты (дефолтный параметр `timeout.reconciliation` в конфигурации `argocd-cm`) контроллер обходит все зарегистрированные Git-репозитории, проверяя наличие новых коммитов. 

* **Минусы пуллинга:** Задержка до 3 минут ломает концепцию мгновенного фидбека (Fast Feedback Loop). Ручное уменьшение интервала опроса в конфигах нагружает API-сервер Git-провайдера и на больших проектах с сотнями репозиториев неизбежно приводит к блокировке по Rate Limits.



#### Решение — Переход на событийную модель (Event-Driven)
Чтобы деплой начинался мгновенно после мерджа кода, на стороне Git-провайдера (GitHub, GitLab, Bitbucket) настраивается **Webhook** на эндпоинт API-сервера ArgoCD.

* **Целевой URL эндпоинта:** `https://<your-argocd-domain>/api/webhook`
* **Тип события (Event Trigger):** Строго `Push` (этого достаточно, так как слияние PR — это тоже пуш в целевую ветку).

#### Как устроен процесс доставки с вебхуком:
1. Разработчик вливает фичу в ветку `main`.
2. GitHub/GitLab мгновенно отправляет POST-запрос на эндпоинт `/api/webhook` ArgoCD с JSON-payload, содержащим информацию о репозитории и хэше нового коммита.
3. ArgoCD перехватывает событие, идентифицирует связанное `Application`, мгновенно сбрасывает для него 3-минутный кэш и запускает цикл реконсиляции немедленно. Время доставки сокращается со 180 секунд до 2-3 мгновений.

#### Безопасность вебхуков (Webhook Secret Tokens)
Держать эндпоинт `/api/webhook` открытым для всего интернета без проверки подлинности — серьезная уязвимость. Злоумышленник может отправлять туда поддельные запросы, заставляя ArgoCD бесконечно опрашивать Git, что вызовет отказ в обслуживании (DoS).

* **Реализация защиты:** В настройках вебхука на GitHub генерируется секретная строка (**Secret Token**). Эта же строка прописывается в Kubernetes-секрет самого ArgoCD (в `argocd-secret` под ключом `webhook.github.secret`).
* **Механизм валидации:** При отправке хука GitHub подписывает тело запроса этим секретом и передает хэш в заголовке `X-Hub-Signature-256`. ArgoCD вычисляет хэш на своей стороне и принимает запрос только при полном совпадении подписей, отсекая любые сторонние манипуляции.
