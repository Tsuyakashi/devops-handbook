## Cloud Fundamentals

### 1. Shared Responsibility Model (Модель разделения ответственности)

Главное правило облачной безопасности: **«Облачный провайдер отвечает за безопасность НАШЕГО облака (Security OF the Cloud), а клиент отвечает за безопасность ВНУТРИ облака (Security IN the Cloud)».**



* **Ответственность провайдера (AWS/GCP/Azure):** Физическая безопасность дата-центров, охлаждение, питание, сетевые кабели, утилизация старых жестких дисков и гипервизоры (виртуализация оборудования). Ты не думаешь о том, как заменить сгоревший блок питания на блейд-сервере.
* **Ответственность DevOps-инженера (Клиента):** * Конфигурация сети (настройка паттернов VPC, файрволов, закрытие лишних портов наружу).
  * Шифрование данных (как в покое — *at rest*, так и при передаче — *in transit*).
  * Менеджмент операционных систем (патчи уязвимостей ядра ОС в EC2-инстансах).
  * IAM (управление доступами, ротация ключей, настройка RBAC).

> **Собес-кейс:** Если злоумышленники получили доступ к базе данных, потому что ты оставил Security Group открытой на `0.0.0.0/0` с дефолтным паролем root — это **твоя** вина. Если дата-центр провайдера сгорел, и данные стерлись, а у тебя не было бэкапов в другом регионе — это **тоже твоя** вина, так как архитектура высокой доступности на стороне клиента.

### 2. Regions & Availability Zones (AZ)

Глобальная инфраструктура облака делится на три уровня: **Regions $\rightarrow$ Availability Zones $\rightarrow$ Edge Locations**.

#### Regions (Регионы)
Географически изолированные области мира (например, `eu-central-1` во Франкфурте или `us-east-1` в Северной Вирджинии).
* Каждая зона полностью независима от другой (свои законы, цены, наборы сервисов).
* **DevOps-критерии выбора региона:** 1. *Compliance / Compliance Laws* (например, GDPR обязывает хранить данные европейских пользователей внутри ЕС).
  2. *Latency* (минимизация пинга — выбираем регион ближе к целевой аудитории).
  3. *Cost* (цена одного и того же инстанса в разных регионах отличается; `us-east-1` обычно самый дешевый, Франкфурт/Сан-Паулу — дороже).

#### Availability Zones (AZ / Зоны доступности)
Один регион состоит из нескольких (минимум трех) изолированных AZ (например, `eu-central-1a`, `eu-central-1b`, `eu-central-1c`).
* **Что такое AZ физически?** Это один или несколько отдельных, физически разнесенных дата-центров с независимым электропитанием, охлаждением и сетевым подключением. Они находятся на расстоянии десятков километров друг от друга на случай локальных катастроф (пожар, наводнение, падение самолета), но соединены между собой сверхбыстрыми оптическими линиями с минимальной задержкой (миллисекунды).
* **Высокая доступность (High Availability):** Архитектура любого production-сервиса (будь то ноды Kubernetes или базы данных) обязана быть **Multi-AZ**. Дизайн-паттерн выглядит так:
  [ВСТАВИТЬ БЛОК КОДА]
  Load Balancer (L7)
    ├── Forward request to App Instance 1 (in AZ-a)
    └── Forward request to App Instance 2 (in AZ-b)
  [КОНЕЦ БЛОКА КОДА]
  Если одна из AZ полностью уйдет в офлайн, балансировщик мгновенно перенаправит 100% трафика на живую ноду в соседней AZ, и пользователи ничего не заметят.

#### Edge Locations (Points of Presence / PoP)
Сотни мелких точек присутствия провайдера по всему миру, которые не являются дата-центрами для вычислений, а служат как CDN (Content Delivery Network — в AWS это сервис *CloudFront*).
* Используются для кэширования статического контента (картинки, JS/CSS, видео) максимально близко к конечному пользователю для снижения задержек и разгрузки основного бэкенда в регионе.

## AWS Core Services: VPC & Networking

### 1. Виртуальное облако (VPC) и подсети (Subnets)

**VPC (Virtual Private Cloud)** — это твоя изолированная логическая сеть внутри инфраструктуры AWS. При создании VPC ты задаешь диапазон IP-адресов в формате CIDR (классический дефолт для проде: `10.0.0.0/16`, что дает 65,536 IP-адресов).

VPC делится на более мелкие куски — **Subnets (Подсети)**, каждая из которых жестко привязана к конкретной **Availability Zone (AZ)**.



#### Public Subnets (Публичные подсети)
* **Что делает её публичной:** Наличие записи в её таблице маршрутизации (Route Table), которая перенаправляет трафик `0.0.0.0/0` на **Internet Gateway (IGW)**.
* **Что там живет:** Публичные балансировщики нагрузки (ALB/NLB), Bastion-хосты (Jump-серверы), NAT-шлюзы.
* **Важно:** Любой инстанс в этой подсети должен иметь Public IP, чтобы отвечать на внешние запросы.

#### Private Subnets (Приватные подсети)
* **Что делает её приватной:** В её Route Table **нет** прямого маршрута к Internet Gateway. Внешний мир из интернета не может напрямую достучаться до хостов внутри этой подсети.
* **Что там живет:** Базы данных (RDS), стейтмент-сервисы, ноды Kubernetes (EKS), очереди сообщений. Everything critical.

---

### 2. Шлюзы: Internet Gateway vs NAT Gateway

| Компонент | Internet Gateway (IGW) | NAT Gateway (NAT-GW) |
| :--- | :--- | :--- |
| **Уровень OSI** | L3 (Маршрутизация) | L3 (Трансляция Source IP) |
| **Где размещается** | Вешается на уровне всего VPC. | Физически создается **внутри Публичной подсети**. |
| **Направление трафика** | **Двунаправленный** (Inbound & Outbound). Позволяет трафику как выходить наружу, так и заходить из интернета. | **Однонаправленный** (Outbound Only). Выпускает трафик наружу, но блокирует любые входящие инициации соединений. |
| **DevOps-применение** | Обеспечивает связь публичных балансеров и инстансов с внешним миром. | Позволяет приватным нодам K8s или серверам БД скачивать обновления, обращаться к сторонним API по HTTPS, но оставаться невидимыми снаружи. |

#### Как устроен Outbound-маршрут для приватной подсети:
[ВСТАВИТЬ БЛОК КОДА]
[Private EC2 Instance] 
       ↓ (Traffic to 0.0.0.0/0)
[Private Route Table] -> Route: 0.0.0.0/0 via nat-xxxxxxxx
       ↓
[NAT Gateway (in Public Subnet)] -> Translates Private IP to NAT's Elastic IP
       ↓
[Public Route Table] -> Route: 0.0.0.0/0 via igw-xxxxxxxx
       ↓
[Internet Gateway] -> Out to the Internet
[КОНЕЦ БЛОКА КОДА]

### 3. Безопасность: Security Groups vs Network ACL (NACL)

Это главный подвох на архитектурных интервью. Запомни: это **два разных уровня защиты**, работающих последовательно.



| Критерий | Security Group (SG) | Network ACL (NACL) |
| :--- | :--- | :--- |
| **Уровень защиты** | **Instance Level**. Работает как файрвол для конкретной сетевой карты инстанса (ENI). | **Subnet Level**. Работает как брандмауэр на границе всей подсети. Пакет сначала проходит через NACL, а потом идет к SG. |
| **Тип (State)** | **Stateful**. Если ты разрешил входящий трафик (Inbound) на порт 80, ответный исходящий трафик (Outbound) выпустится автоматически, независимо от правил Outbound. | **Stateless**. Каждое правило проверяется отдельно. Если разрешил Inbound на порт 80, обязан явно написать правило на Outbound для эфемерных портов (`1024-65535`), иначе ответ не уйдет. |
| **Типы правил** | Поддерживает только **Allow-правила** (Разрешить). Всё, что не разрешено явно — заблокировано. | Поддерживает и **Allow**, и **Deny** (Запретить) правила. Правила обрабатываются по порядку номеров (Rule #). |
| **Таргетинг** | Можно указывать в качестве источника/назначения **другую Security Group** (удобно для микросервисов). | Можно указывать только **CIDR-блоки** (диапазоны IP-адресов). |

#### Реальный DevOps-сценарий (Паттерн применения):
* Ты используешь **Security Groups** для повседневного менеджмента доступов: например, разрешаешь своей SG для бэкенда принимать трафик только со стороны SG балансировщика.
* Ты используешь **NACL** для экстренных ситуаций или глобальных ограничений: например, если твой сервис прямо сейчас атакует конкретная китайская или ботнет-сеть, ты заходишь в NACL подсети и вешаешь правило `#100: DENY [IP_АТАКУЮЩЕГО]/24`. Трафик заблокируется на подлете к подсети, даже не нагружая CPU твоих инстансов.

## AWS Core Services: Compute & Auth

### 1. Вычислительные мощности: EC2 (Elastic Compute Cloud)

**EC2** — это виртуальные серверы (инстансы) в облаке AWS, работающие на базе кастомного гипервизора (сейчас это AWS Nitro на базе KVM).

* **Instance Types (Типы инстансов):** Делятся на семейства под разные задачи:
  * `General Purpose` (семейства `t` и `m`): Сбалансированные ресурсы. Семейство `t` (например, `t3.medium`) использует концепт **CPU Credits** — они подходят для разработки и стейджингов, где нагрузка идет всплесками.
  * `Compute Optimized` (семейство `c`): Высокая частота CPU. Идеально для воркеров, кодирования видео и высоконагруженных веб-серверов.
  * `Memory Optimized` (семейство `r`): Огромный объем RAM. Для in-memory баз данных (Redis) и тяжелых СУБД (PostgreSQL).
* **Purchasing Options (Модели оплаты):**
  * `On-Demand`: Платишь посекундно за то, что запущено. Самый дорогой вариант, без обязательств.
  * `Savings Plans / Reserved Instances`: Контракт на 1-3 года со скидкой до 72%. Подходит для стабильной, предсказуемой базовой нагрузки.
  * `Spot Instances`: AWS продает свои избыточные (пустующие) мощности со скидкой до 90%. **Главный нюанс:** AWS может забрать этот инстанс обратно в любой момент, предупредив всего за 2 минуты.
  * *DevOps-паттерн:* Споты идеально подходят для stateless-нод в кластерах Kubernetes (EKS), CI/CD воркеров (GitLab runners) и любых систем пакетной обработки данных, которые умеют безболезненно перезапускаться.

### 2. Безопасность и доступы: Компоненты IAM

**IAM (Identity and Access Management)** — это глобальный (не привязанный к регионам) сервис управления доступом к ресурсам AWS.



#### Users (Пользователи)
Конкретный человек или сущность (например, Jenkins-сервер старой конфигурации). Имеет статические учетные данные: пароль от консоли или пару **Access Key ID / Secret Access Key** для работы через CLI/API.
* *Production Anti-pattern:* Никогда не создавай IAM-пользователей для приложений и не зашивай их Access-ключи в конфиги кода или переменные GitLab/GitHub Actions. Ключи имеют свойство утекать на GitHub, что ведет к моментальному взлому аккаунта ботами.

#### Groups (Группы)
Коллекция IAM-пользователей. Разрешения вешаются на группу, а не на отдельных людей (например, группа `developers` имеет доступ к Read-Only ресурсам, а группа `devops` — полный административный доступ).

#### Policies (Политики)
JSON-документы, которые явно описывают, **КТО, ЧТО и НАД КАКИМИ** ресурсами может делать. Политики бывают:
* `AWS Managed`: Дефолтные политики от самого AWS (например, `AdministratorAccess`).
* `Customer Managed`: Твои собственные кастомные и максимально гранулярные политики (принцип Least Privilege).

Структура JSON-политики состоит из блоков **Statement**:
[ВСТАВИТЬ БЛОК КОДА]
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",          # Разрешить или Запретить (Allow/Deny)
            "Action": [
                "s3:GetObject",         # Какое действие с сервисом совершается
                "s3:ListBucket"
            ],
            "Resource": [
                "arn:aws:s3:::my-app-bucket",       # Конкретный ресурс (ARN)
                "arn:aws:s3:::my-app-bucket/*"
            ]
        }
    ]
}
[КОНЕЦ БЛОКА КОДА]

### 3. Главный концепт облачной безопасности: IAM Roles & AssumeRole

#### IAM Roles (Роли)
В отличие от пользователя, **Роль не имеет постоянных паролей или Access-ключей**. Роль — это идентичность с набором прав, которую может временно «надеть» на себя любая легитимная сущность: инстанс EC2, под в Kubernetes (через IRSA) или внешний пайплайн (через OIDC).

Когда EC2-инстанс надевает роль (через механизм Instance Profile), специальный внутренний сервис AWS (Instance Metadata Service — IMDS) генерирует для него **временные access-ключи**, которые автоматически ротируются каждые несколько часов. Если злоумышленник взломает твой сервер, он найдет в памяти только временные ключи, которые превратятся в тыкву через короткое время.

#### Концепт AssumeRole (Принятие роли)
Это процесс, при котором одна сущность временно запрашивает права другой роли через системный вызов API `sts:AssumeRole` (Security Token Service).



**Как это работает на практике (Архитектурный паттерн Cross-Account Access):**
У тебя в компании есть два аккаунта AWS: `Management-Account` (где сидят пользователи/админы) и `Production-Account` (где крутятся боевые базы данных и сервисы).

1. В `Production-Account` создается IAM Роль `ProdDBAdminRole`. 
2. В этой роли прописывается **Trust Relationship Policy** (Политика доверия). Она указывает: *«Я доверяю аккаунту Management-Account и разрешаю пользователям оттуда принимать меня»*.
3. Твой DevOps-инженер логинится в свой safe-юзер в `Management-Account`, делает вызов `AssumeRole` в сторону `ProdDBAdminRole`.
4. Сервис AWS STS проверяет права, одобряет запрос и возвращает инженеру краткосрочные токены. Инженер заходит в прод-базу, делает работу, после чего токен протухает. **Никаких общих паролей и постоянных ключей доступа между аккаунтами не существует.**

## AWS Core Services: Storage & DB

### 1. Объектное хранилище: S3 (Simple Storage Service)

**AWS S3** — это высокомасштабируемое и отказоустойчивое объектное хранилище (не путать с классической файловой системой POSIX, здесь нет структуры папок — всё является объектом с уникальным ключом-путем). Данные хранятся в корзинах — **Buckets**.

#### Безопасность: Bucket Policies vs IAM Policies
* **IAM Policies:** Привязываются к *сущностям* (Users, Roles) внутри твоего аккаунта: «Роли `AppRole` разрешено читать из бакета `my-bucket`».
* **Bucket Policies:** Привязываются напрямую к самому *бакету* (Resource-based policy). Позволяют управлять доступом извне.
  * *DevOps-применение:* Bucket Policies незаменимы для Cross-Account доступа (разрешить аккаунту Б читать логи из аккаунта А) или для настройки публичного доступа (например, открыть статику сайта для всего интернета):
    [ВСТАВИТЬ БЛОК КОДА]
    {
        "Version": "2012-10-17",
        "Statement": [
            {
                "Sid": "PublicReadGetObject",
                "Effect": "Allow",
                "Principal": "*",
                "Action": "s3:GetObject",
                "Resource": "arn:aws:s3:::my-public-bucket/*"
            }
        ]
    }
    [КОНЕЦ БЛОКА КОДА]

#### Оптимизация затрат: Lifecycle Rules (Жизненные циклы)
Хранить терабайты логов или старых бэкапов на стандартном "горячем" диске S3 Standard — неоправданно дорого. Для этого настраивают автоматические **Lifecycle Rules**, которые перемещают объекты между классами хранения в зависимости от их возраста:

1. `S3 Standard` — Высокая стоимость хранения, моментальный доступ (для активных данных).
2. `S3 Standard-IA` (Infrequent Access) — Хранение дешевле, но берется плата за каждый гигабайт считанных данных. Для бэкапов, к которым обращаются раз в месяц.
3. `S3 Glacier Flexible/Deep Archive` — Копеечное хранение архивных данных. Извлечение занимает от нескольких минут до 12 часов. Идеально для хранения старых логов по требованию регуляторов (Compliance).

*Типичный DevOps-пайплайн:* Объект создается в `Standard` $\rightarrow$ через 30 дней падает в `Standard-IA` $\rightarrow$ через 90 дней улетает в `Glacier Deep Archive` $\rightarrow$ через 365 дней безвозвратно удаляется.

### 2. Базы данных: RDS (Relational Database Service)

**RDS** — это управляемый (Managed) сервис реляционных баз данных (PostgreSQL, MySQL, MariaDB). AWS берет на себя бэкапы, обновление минорных версий ОС и СУБД, а также масштабирование дисков.

На архитектурных интервью критически важно понимать разницу между двумя концепциями масштабирования и надежности: **Multi-AZ Deployment** и **Read Replicas**.



| Критерий | Multi-AZ (Высокая доступность) | Read Replicas (Масштабирование) |
| :--- | :--- | :--- |
| **Основная цель** | **Disaster Recovery (Отказоустойчивость)**. Защита данных от падения дата-центра. | **Scalability (Масштабируемость)**. Снятие нагрузки на чтение (Read-heavy приложения) с основного мастера. |
| **Репликация** | **Синхронная**. Запись считается успешной только тогда, когда данные физически записались на мастер в AZ-1 и на Standby в AZ-2. Нулевая потеря данных (RPO=0). | **Асинхронная**. Мастер записывает данные себе и фоном отправляет их на реплики. Возможен небольшой лаг репликации (*Replication Lag*). |
| **Режим работы** | Активен только Мастер. Standby-инстанс «спит» (инструменты/приложения не могут слать туда даже SELECT-запросы). | Все реплики доступны для чтения (`Read-Only`). Приложение распределяет SELECT-запросы на них. |
| **Поведение при аварии (Failover)** | **Автоматический**. Если мастер умер, AWS за ~60-120 секунд переключает DNS-запись базы на Standby-ноду. Приложению не нужно менять IP-адреса в конфигах подключения. | **Ручной/Программный**. Если мастер умирает, одна из реплик должна быть вручную (или скриптом) повышена до статуса нового Мастера (*Promote to Master*). |
| **География** | Строго внутри одного региона (разные AZ). | Могут быть распределены как внутри региона, так и находиться в **других регионах** (Cross-Region Replicas) для ускорения работы глобальных приложений. |

#### Сводный DevOps-инсайт для дизайна систем:
> «Если тебе нужна база, которая выживет при падении метеорита на дата-центр Франкфурта — включай **Multi-AZ**. Если твоё Node.js приложение ложится от сотен тысяч аналитических SELECT-запросов пользователей — создавай пул **Read Replicas** и перенаправляй легкие запросы на чтение туда».

## AWS Core Services: Orchestration & Monitoring

### 1. Управляемый Kubernetes: EKS (Elastic Kubernetes Service)

**AWS EKS** — это托管 (Managed) сервис Kubernetes, где AWS полностью берет на себя развертывание, масштабирование и обеспечение высокой доступности компонентов **Control Plane** (API Server, etcd, scheduler, controller manager) в режиме Multi-AZ. Ты управляешь только рабочими нодами (**Worker Nodes**).



#### Рабочие ноды: Managed Node Groups vs Fargate
Для запуска контейнеров (подов) в EKS используются два подхода:
* **Managed Node Groups:** AWS автоматически создает и настраивает EC2-инстансы, объединяет их в Auto Scaling Group (ASG), обновляет на них версию ОС/Kubernetes и управляет жизненным циклом. Ты платишь за стандартные виртуальные машины EC2.
* **AWS Fargate:** Serverless-подход для контейнеров. Под каждый под динамически выделяется изолированная виртуальная микро-машина прямо в инфраструктуре AWS. Нет необходимости администрировать EC2 и настраивать масштабирование нод. Ты платишь только за реальные CPU/RAM, потребляемые запущенными подами.

#### Безопасность подов: IRSA (IAM Roles for Service Accounts)
* *Проблема:* Как дать поду внутри Kubernetes (например, бэкенду на Node.js) права на загрузку картинок в бакет AWS S3?
* *Устаревший небезопасный путь:* Повесить IAM-роль на саму EC2-ноду (worker node). 
  * *В чем минус:* **Все поды**, запущенные на этой ноде (даже соседние чужие микросервисы), получат полный доступ к этому бакету S3. Это нарушает принцип Least Privilege.
* *Современный облачный стандарт:* **IRSA**. Механизм связывает стандартный `ServiceAccount` в Kubernetes с конкретной `IAM Role` в AWS через OpenID Connect (OIDC).



**Как устроен процесс авторизации через IRSA:**
1. В AWS настраивается Identity Provider (IdP) связывающий твой EKS-кластер и IAM.
2. Создается IAM-роль, в политике доверия (Trust Policy) которой разрешено принимать её только конкретному ServiceAccount из конкретного нэймспейса K8s.
3. В манифесте Kubernetes ты просто вешаешь аннотацию на ServiceAccount:
   [ВСТАВИТЬ БЛОК КОДА]
   apiVersion: v1
   kind: ServiceAccount
   metadata:
     name: my-app-sa
     namespace: default
     annotations:
       eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/MyS3AccessRole
   [КОНЕЦ БЛОКА КОДА]
4. Мутирующий вебхук EKS автоматически прокидывает внутрь пода AWS токен и переменные окружения. Приложение под капотом использует AWS SDK, прозрачно запрашивая временные доступы у AWS STS. Доступ изолирован на уровне конкретного контейнера.

---

### 2. Мониторинг и обсервабилити: CloudWatch

**AWS CloudWatch** — это комбайн для сбора метрик, логов и событий со всей твоей инфраструктуры AWS.

* **CloudWatch Metrics:** Сервис автоматически собирает базовые метрики со всех ресурсов (загрузка CPU инстансов EC2, количество свободных портов на NAT Gateway, объем Replication Lag на репликах RDS). Метрики хранятся по умолчанию и используются для построения графиков.
* **CloudWatch Alarms (Алерты):** Позволяет настроить триггеры на основе метрик.
  * *Пример применения:* Если средняя загрузка CPU в Auto Scaling Group превышает 75% в течение 5 минут $\rightarrow$ CloudWatch дергает триггер на добавление новой EC2-ноды в кластер, а также отправляет оповещение инженерам через SNS (Simple Notification Service) в Slack/Telegram.
* **CloudWatch Logs:** Централизованное хранилище логов.
  * Туда стекаются логи самой ОС (через CloudWatch Logs Agent), логи API-сервера Kubernetes (EKS Control Plane Logs), логи с балансировщиков и AWS Lambda функций.
  * Поддерживает **Metric Filters** — возможность парсить текстовые логи «на лету» регулярными выражениями. Например, можно находить в логах бэкенда слово `ERROR` и автоматически превращать его частоту в числовую метрику, на которую затем повесить Alarm.

На архитектурных интервью часто можно услышать вопрос: *«Мы мигрируем этот проект из AWS в Google Cloud (или Azure). Какие аналоги сервисов вы выберете для построения сети, вычислений и оркестрации контейнеров?»*

### Ключевые DevOps-нюансы при кросс-клаудном проектировании

#### 1. Сетевая модель: AWS/Azure vs GCP
* В **AWS (VPC)** и **Azure (VNet)** сеть создается в рамках **одного конкретного региона**. Чтобы разделить её на подсети в разных Availability Zones, ты явно нарезаешь CIDR-блоки под каждую AZ.
* В **GCP (VPC Network)** сеть по умолчанию является **глобальной (Global)**. Это значит, что одна и та же VPC-сеть прозрачно объединяет подсети, созданные в абсолютно разных регионах мира (например, Франкфурт и Сингапур). Маршрутизация между регионами внутри сети провайдера работает "из коробки" без дополнительных VPN-туннелей или пирингов.

#### 2. Сервисные аккаунты вместо Ролей
* В AWS для предоставления прав инстансу или поду используется механизм временного принятия роли (`AssumeRole`).
* В GCP и Azure концепт немного отличается. Там создается специальная сущность — **Service Account** (в GCP) или **Managed Identity** (в Azure). Это, по сути, "цифровой паспорт" для приложения, который вешается на виртуальную машину или под Kubernetes, наделяя их правами без использования постоянных секретных ключей.

#### 3. Специфика Kubernetes (EKS vs GKE vs AKS)
* **GKE (Google)** исторически считается эталоном среди управляемых Kubernetes-кластеров, так как сам Kubernetes был создан внутри Google. GKE обновляется быстрее всех, имеет лучшую нативную интеграцию с экосистемой и мощнейший автоскейлинг нод (*GKE Autopilot*).
* **EKS (AWS)** требует больше ручной настройки "руками" (интеграция OIDC для IRSA, установка драйверов дисков CSI, настройка сетевого плагина AWS-VPC CNI).
* **AKS (Azure)** предлагает отличную бесшовную интеграцию с Active Directory (Entra ID) для enterprise-компаний, где права на деплой в кластер завязаны на корпоративные учетные записи сотрудников.

## FinOps & Cloud Architecture

### 1. Концепция FinOps (Financial Operations)

**FinOps** — это культурная практика и методология управления облачными затратами, объединяющая инженеров, финансистов и бизнес. Главная цель — не просто бездумно урезать бюджеты, а максимизировать ценность каждого потраченного доллара в облаке.



#### Три фазы жизненного цикла FinOps:
1. **Inform (Информирование):** Обеспечение полной прозрачности затрат. Каждая команда должна точно знать, сколько стоят её сервисы.
   * *Инструмент DevOps:* Разработка жесткой **Tagging Policy** (Политики тегирования). Все ресурсы в облаке должны быть размечены тегами (например, `Environment: Prod`, `Project: AgenticNews`, `Owner: DevOpsTeam`). Без тегов невозможно понять, какой именно микросервис сжирает бюджет.
2. **Optimize (Оптимизация):** Поиск и устранение скрытых и неэффективных трат (Waste Reduction).
3. **Operate (Управление):** Настройка непрерывного мониторинга бюджетов и автоматических алертов на аномалии в затратах.

#### Главные DevOps-метрики для оптимизации стоимости:
* **Rightsizing (Оптимизация размеров):** Анализ реальной утилизации ресурсов. Если инстанс базы данных RDS загружен по CPU всего на 5%, а по памяти на 10%, его необходимо даунгрейднуть на более дешевый тип.
* **Удаление "сиротских" (Orphaned) ресурсов:** Поиск забытых ресурсов, за которые продолжает капать счет:
  * Непривязанные публичные IP (в AWS зарезервированный, но не используемый Elastic IP стоит денег).
  * Диски виртуальных машин (EBS Volumes), которые остались жить после удаления самих инстансов EC2.
  * Старые бэкапы и недогруженные многокомпонентные объекты в S3 (настраивается через abort incomplete multipart uploads в Lifecycle Rules).

---

### 2. Spot Instances в Enterprise-архитектуре

Использование прерываемых мощностей (AWS Spot Instances, GCP Spot VMs, Azure Spot VMs) со скидкой до 90% — самый мощный рычаг экономии. Однако он требует проектирования **Cloud-Native / Fault-Tolerant** архитектуры.

#### Стратегии выживания приложений на спотах:
* **Stateless Only:** На спотовых нодах можно запускать только те приложения, которые не хранят состояние внутри себя и могут быть безболезненно убиты в любую секунду (веб-бэкенды, API-шлюзы, кэширующие прокси). Базы данных, стейтфул-очереди сообщений и сессии пользователей туда селить **строго запрещено**.
* **Обработка прерывания (Spot Interruption Handling):** Облачные провайдеры предупреждают о досрочном изъятии инстанса заранее (AWS за 2 минуты, GCP за 30 секунд). 
  * *Реализация в K8s:* В кластере разворачивается специальный демон (например, *AWS Node Termination Handler*). Как только поступает сигнал об изъятии спот-машины, демон мгновенно вешает на неё лейбл `NoSchedule` и делает `kubectl drain`, плавно перенося все запущенные поды на другие, живые ноды.
* **Mixed Node Groups (Гибридные группы):** Паттерн, при котором Auto Scaling Group собирается в пропорции. Например, 30% — стабильные On-Demand машины (базовый уровень, который гарантирует, что прод не упадет полностью), а 70% — дешевые Spot-машины для масштабирования под пиковую нагрузку.

---

### 3. Мультиклаудная сеть (Multi-cloud Networking)

Когда бизнес принимает решение развернуть часть инфраструктуры в AWS (например, основное приложение), а часть в GCP (например, специфичные MLOps-сервисы и BigData-аналитику), встает задача организации безопасного сетевого туннеля между разными облаками.



#### Варианты построения Cross-Cloud сети (от простых к надежным):

* **Site-to-Site VPN (IPsec VPN):** Самый быстрый и дешевый вариант. Между публичными шлюзами AWS (Virtual Private Gateway) и GCP (Cloud VPN) поднимается шифрованный туннель через общественный интернет.
  * *Минусы:* Трафик идет через публичные магистрали интернета, из-за чего пинг (latency) может плавать, а пропускная способность ограничена возможностями VPN-шлюзов.
* **Выделенные каналы (AWS Direct Connect + GCP Interconnect):** Провайдеры позволяют протянуть физический кабель от твоей стойки в ЦОД напрямую к своим точкам присутствия. При мультиклауде они соединяются через независимых сетевых провайдеров (Equinix, Megaport).
  * *Плюсы:* Гарантированная минимальная задержка (миллисекунды), выделенная полоса пропускания (10–100 Гбит/с), максимальная безопасность.
  * *Минусы:* Очень высокая стоимость и долгое время подключения.
* **Overlay Networks (Программные SDN сети):** Построение виртуальной оверлейной сети поверх облаков с помощью mesh-инструментов, таких как WireGuard, HashiCorp Consul или специализированных решений (Aviatrix). Позволяет абстрагироваться от сетевых настроек конкретного провайдера.
