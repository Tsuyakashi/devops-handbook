## Relational Databases

### 1. PostgreSQL vs MySQL/MariaDB: когда что выбирать

Оба — зрелые open-source RDBMS с полной поддержкой ACID, но исторически разошлись по архитектурным акцентам.

| Критерий | PostgreSQL | MySQL / MariaDB |
| :--- | :--- | :--- |
| **Архитектура процессов** | Process-per-connection (каждое соединение — отдельный процесс ОС). Тяжелее по памяти на соединение, отсюда обязательный **PgBouncer** в проде. | Thread-per-connection. Легче по памяти на соединение "из коробки". |
| **Типы данных** | Богатый набор: `JSONB` (бинарный, индексируемый JSON), массивы, `hstore`, кастомные типы, `UUID` нативно. | Более скромный набор; JSON есть, но исторически хуже индексировался (до MySQL 8 / MariaDB 10.x). |
| **Хранимые движки** | Единый движок хранения. | **Pluggable storage engines** — `InnoDB` (транзакционный, дефолт) vs `MyISAM` (legacy, без транзакций, но быстрее на чтение при read-only нагрузке). |
| **Расширяемость** | Экосистема расширений (`PostGIS` для геоданных, `pg_partman` для партиционирования, `TimescaleDB` для time-series). | Расширяется слабее, но проще в администрировании для типовых веб-нагрузок. |
| **Типичный DevOps-выбор** | Сложная бизнес-логика, аналитика, геоданные, когда важна строгая консистентность и богатые типы. | Классический LAMP/веб-стек, где важна простота репликации и низкий порог входа. |

> **DevOps-инсайт:** На практике выбор чаще определяется экосистемой фреймворка (Prisma/Django тяготеют к Postgres, WordPress/Laravel — к MySQL), а не голым перформансом — обе БД на современном железе покрывают 95% нагрузок веб-приложений без проблем.

### 2. Оптимизация запросов: `EXPLAIN ANALYZE` и индексы

Главный инструмент диагностики медленного запроса — план выполнения. Без него любая оптимизация — гадание.

```sql
EXPLAIN ANALYZE
SELECT * FROM orders WHERE customer_id = 42 AND status = 'shipped';
```

Ключевые сигналы в выводе, на которые смотрит DevOps/DBA:

* **`Seq Scan` (Sequential Scan)** на большой таблице — БД читает **всю** таблицу целиком, строка за строкой, вместо использования индекса. Красный флаг для таблиц от сотен тысяч строк.
* **`Index Scan` / `Index Only Scan`** — запрос использует индекс. `Index Only Scan` — лучший случай: все нужные данные есть прямо в индексе, обращения к самой таблице (heap) не требуется.
* **`rows=` (estimated) vs `actual rows=`** — большое расхождение между оценкой планировщика и реальным количеством строк сигнализирует об устаревшей статистике таблицы (нужен `ANALYZE table_name`).

#### Индексы: не панацея, а компромисс

```sql
-- Обычный B-Tree индекс — дефолт для большинства случаев (равенство, диапазоны, сортировка)
CREATE INDEX idx_orders_customer_id ON orders (customer_id);

-- Составной индекс — порядок колонок критичен: под запросы вида WHERE a = ? AND b = ?
CREATE INDEX idx_orders_customer_status ON orders (customer_id, status);

-- Частичный индекс — индексируем только "горячее" подмножество строк, экономим место и I/O
CREATE INDEX idx_orders_pending ON orders (customer_id) WHERE status = 'pending';
```

* **Цена индекса:** Каждый `INSERT`/`UPDATE`/`DELETE` обязан обновить **все** индексы таблицы, а не только данные. Таблица с 10 индексами пишется заметно медленнее, чем без них — индексы ускоряют `SELECT` за счёт замедления записи.
* **Правило составного индекса (Leftmost Prefix):** Индекс `(customer_id, status)` эффективно используется для запросов по `customer_id` и по `(customer_id, status)`, но **не** используется для запроса только по `status` — колонки должны совпадать с индексом слева направо.

#### N+1 Problem

Классическая ловушка ORM (Prisma, Sequelize, SQLAlchemy): вместо одного JOIN-запроса приложение делает 1 запрос на список + N запросов на каждую связанную сущность.

```text
-- Плохо: 1 запрос на список заказов + N запросов на клиента каждого заказа
SELECT * FROM orders;                          -- 1 запрос, вернул 100 строк
SELECT * FROM customers WHERE id = ?;          -- ×100 запросов в цикле

-- Хорошо: один JOIN или eager loading на уровне ORM
SELECT o.*, c.* FROM orders o
JOIN customers c ON c.id = o.customer_id;      -- 1 запрос
```

> **DevOps-инсайт:** N+1 почти невозможно поймать по логам приложения — он проявляется как аномально высокий `rate(pg_stat_activity)` или счётчик запросов в APM (New Relic/Datadog) при абсолютно нормальном коде на вид. Ищется через `slow query log` или `pg_stat_statements`, где один и тот же шаблон запроса встречается сотнями за секунду.

### 3. Стратегии бэкапов

| Тип бэкапа | Механизм | Плюсы | Минусы |
| :--- | :--- | :--- | :--- |
| **Logical (`pg_dump` / `mysqldump`)** | Экспорт данных в SQL-команды или собственный формат (`pg_dump -Fc`). | Портируем между версиями и даже СУБД, можно восстановить отдельную таблицу. | Медленно на больших базах (десятки/сотни ГБ), нагружает CPU при снятии, не даёт точку восстановления "на секунду". |
| **Physical (`pg_basebackup`, файловая копия `/var/lib/mysql`)** | Побайтовая копия файлов данных на диске. | Быстрое снятие и восстановление на больших объёмах. | Жёстко привязан к той же мажорной версии СУБД и архитектуре, нельзя восстановить частично. |
| **Continuous Archiving (WAL / binlog)** | Непрерывная отправка журнала транзакций (**WAL** в Postgres, **binlog** в MySQL) в отдельное хранилище (S3). | **Point-in-Time Recovery (PITR)** — восстановление на любую секунду между полными бэкапами, включая "за минуту до дропа таблицы". | Требует физического полного бэкапа как базовой точки + непрерывного потока WAL/binlog поверх него. |

#### PITR: восстановление "за секунду до инцидента"

Самый частый DevOps-кейс на реальном проде: разработчик выполнил `DELETE FROM users;` без `WHERE`. Обычный ежедневный `pg_dump` за прошлую ночь потеряет весь день данных. PITR решает это иначе:

```text
[Полный физический бэкап в 00:00] → [непрерывный поток WAL-файлов в S3] → [DELETE случился в 14:32:07]
                                                                                    ↓
                            Восстановление: базовый бэкап + WAL до 14:32:06 (за секунду ДО инцидента)
```

```bash
# Postgres: восстановление до конкретного timestamp через recovery_target_time
restore_command = 'aws s3 cp s3://backups/wal/%f %p'
recovery_target_time = '2026-06-30 14:32:06'
```

> **Правило 3-2-1:** 3 копии данных, на 2 разных типах носителей, 1 копия — географически в другом регионе/датацентре (см. концепцию Lifecycle Rules и Cross-Region репликации в [Cloud Providers](../AWS-and-Cloud/README.md#aws-storage-db)). Бэкап, лежащий в том же дата-центре, что и прод — не бэкап, а иллюзия бэкапа на случай отказа одного диска, но не на случай пожара в серверной.

#### Проверка бэкапов (Restore Testing)

Бэкап, который ни разу не восстанавливали тестово — это бэкап Шрёдингера: неизвестно, рабочий он или битый, пока не понадобится в 3 часа ночи во время реального инцидента. DevOps-практика: автоматический CI-джоб, который раз в неделю разворачивает последний бэкап на чистом инстансе и гоняет `SELECT count(*)` по ключевым таблицам.

### 4. Master-Slave репликация

#### Механизм: Streaming Replication (Postgres) / Binlog Replication (MySQL)

```text
[Master/Primary]  →  пишет изменения в WAL (Postgres) / binlog (MySQL)
        ↓ (передаёт поток изменений по сети)
[Slave/Replica]  →  применяет тот же поток изменений локально, оставаясь read-only
```

* **Master (Primary):** Принимает **все** операции записи (`INSERT`/`UPDATE`/`DELETE`). Единственный источник истины.
* **Slave (Replica):** Read-only копия, непрерывно применяющая изменения из потока Master'а. Может обслуживать `SELECT`-запросы, снимая нагрузку с Primary — тот же паттерн Read Replicas, что разобран для управляемого RDS в [Cloud Providers](../AWS-and-Cloud/README.md#aws-storage-db), только здесь настраивается вручную на своих серверах, а не кнопкой в консоли AWS.

#### Синхронная vs Асинхронная репликация

| Критерий | Синхронная (Sync) | Асинхронная (Async) |
| :--- | :--- | :--- |
| **Подтверждение записи** | Master ждёт подтверждения от Slave, что данные физически записаны, прежде чем сказать клиенту "commit успешен". | Master коммитит локально и сразу отвечает клиенту, репликация идёт в фоне. |
| **Риск потери данных** | Нулевой (RPO=0) — если Master упал сразу после commit, Slave точно имеет эти данные. | Есть **Replication Lag** — при падении Master'а последние N миллисекунд/секунд транзакций могут быть потеряны. |
| **Влияние на latency записи** | Каждая запись ждёт сетевой round-trip до реплики — заметно медленнее. | Latency записи как у одиночной БД — Slave не в критическом пути. |
| **Типичный выбор в проде** | Финансовые транзакции, где потеря данных недопустима. | Подавляющее большинство веб-приложений — async с мониторингом lag'а. |

#### Failover: ручной vs автоматический

В отличие от управляемого RDS (см. [Cloud Providers](../AWS-and-Cloud/README.md#aws-storage-db)), где AWS сам переключает DNS за 60-120 секунд, самостоятельно поднятая репликация **не** умеет самостоятельно повышать Slave до Master при падении Primary — это либо ручная операция (`pg_promote`), либо задача внешнего инструмента:

* **Patroni** (Postgres) — HA-обвязка на базе distributed consensus (etcd/Consul/ZooKeeper), автоматически детектирующая падение Primary и промоутящая наиболее актуальную реплику. Концептуально похоже на связку `keepalived` + `kubeadm` из HA control plane этого хендбука (см. [`ansible/templates/keepalived.conf.j2`](../../ansible/templates/keepalived.conf.j2)) — тот же принцип "виртуальный лидер + автоматическое переключение при недоступности".
* **Orchestrator** (MySQL) — аналогичный инструмент для топологий MySQL/MariaDB, умеет строить и визуализировать сложные многоуровневые репликации.

> **Собес-кейс:** Почему нельзя просто "включить два Master'а и писать в оба для надёжности" (Multi-Master)? Потому что при конфликте записи одних и тех же строк с двух сторон возникает **split-brain** — какая версия правильная, если оба Master'а считают себя источником истины? Решение существует (`Galera Cluster` для MySQL с синхронной multi-master репликацией через сертификационный конфликт-контроль), но требует принципиально другой архитектуры, а не "просто два обычных Postgres, пишущих друг в друга".

## NoSQL Storage

### 1. MongoDB: документный движок

MongoDB хранит данные не строками таблиц, а **документами** в формате, похожем на JSON (физически — бинарный **BSON**), сгруппированными в **коллекции** (аналог таблиц, но без фиксированной схемы).

```json
// Документ в коллекции "orders" — вложенная структура без JOIN
{
  "_id": ObjectId("64f1a2b3c4d5e6f7a8b9c0d1"),
  "customer": { "name": "Иван Петров", "email": "ivan@example.com" },
  "items": [
    { "sku": "SKU-001", "qty": 2, "price": 499 },
    { "sku": "SKU-002", "qty": 1, "price": 1200 }
  ],
  "status": "shipped",
  "created_at": ISODate("2026-06-15T10:00:00Z")
}
```

#### Schema-less vs Schema-flexible

MongoDB часто называют "безсхемным", но точнее — **schema-flexible**: схема не enforced на уровне СУБД (можно вставить документ с произвольным набором полей), но приложение всё равно неявно ожидает определённую структуру. На практике это компенсируется:

* **Валидацией на уровне приложения** (Mongoose/Prisma-схемы).
* **`$jsonSchema` валидатором** — MongoDB умеет опционально навязывать структуру на уровне коллекции, если команде нужна строгость без перехода на RDBMS.

#### Когда MongoDB — правильный выбор

* **Меняющаяся, вложенная структура данных**, которую больно раскладывать по нормализованным таблицам (каталоги товаров с произвольными атрибутами у разных категорий, контент CMS).
* **Горизонтальное масштабирование записи "из коробки"** через **Sharding** — данные автоматически распределяются по нескольким серверам (шардам) по shard key, в отличие от классической RDBMS, где шардирование — ручная и болезненная архитектурная задача.
* **Read-heavy паттерн с денормализацией:** Если приложение почти всегда читает заказ целиком со всеми товарами (как в примере выше), хранение их одним документом устраняет JOIN'ы полностью — вся нужная информация уже лежит рядом физически.

#### Репликация: Replica Set

MongoDB реализует высокую доступность через **Replica Set** — группу из нечётного числа узлов (обычно 3), где один узел — `PRIMARY` (принимает записи), остальные — `SECONDARY` (реплицируют асинхронно и могут отдавать чтение). При падении Primary узлы **автоматически** проводят выборы нового лидера (`election`) — тот же принцип quorum-консенсуса, что и у etcd в Control Plane Kubernetes (см. [Containerization & Orchestration](../Containerization-and-Orchestration/README.md#kubernetes)), только реализованный внутри самой СУБД, без внешнего Patroni-подобного инструмента.

### 2. Redis: in-memory кэш и хранение сессий

Redis — key-value хранилище, живущее **полностью в оперативной памяти**, что даёт задержки операций в микросекундах против миллисекунд у дисковых СУБД.

#### Структуры данных (не просто key-value)

В отличие от простого кэша (Memcached), Redis предоставляет богатые типы значений:

| Тип | Пример команды | Типичное применение |
| :--- | :--- | :--- |
| **String** | `SET session:abc123 "user_data" EX 3600` | Кэш ответа API, простые счётчики (`INCR`) |
| **Hash** | `HSET user:42 name "Иван" age 30` | Хранение объекта без сериализации всего JSON целиком |
| **List** | `LPUSH queue:emails "task1"` | Простая очередь FIFO (без гарантий Kafka/RabbitMQ, но для лёгких задач достаточно) |
| **Set / Sorted Set** | `ZADD leaderboard 100 "player1"` | Лидерборды, уникальные множества, ранжирование по score |
| **Pub/Sub** | `PUBLISH channel:notifications "..."` | Простая событийная шина в реальном времени (без персистентности сообщений — если подписчика нет онлайн, сообщение теряется навсегда) |

#### Персистентность: RDB vs AOF

Redis — in-memory, но данные можно (и нужно в проде) сохранять на диск, чтобы пережить рестарт процесса.

* **RDB (Redis Database Snapshot):** Периодический полный снимок памяти на диск (например, раз в 5 минут при условии N изменений). Быстрое восстановление, но при падении между снимками теряются последние минуты данных.
* **AOF (Append Only File):** Журнал **каждой** операции записи, применяемый последовательно (концептуально аналог WAL в Postgres / binlog в MySQL из раздела [Relational Databases](#rdbms) выше). Минимальная потеря данных при падении, но файл растёт быстрее и требует периодического `BGREWRITEAOF` для сжатия.
* **Комбинация RDB + AOF** — типичный прод-выбор: быстрый RDB для полных снимков + AOF поверх для минимизации окна потери данных.

#### Eviction Policies: что происходит, когда память заканчивается

Redis ограничен физической RAM (`maxmemory` в конфиге). При достижении лимита включается политика вытеснения:

* **`noeviction`** — новые записи отклоняются с ошибкой (безопасно, но ломает приложение при исчерпании памяти).
* **`allkeys-lru`** — вытесняются наименее недавно использованные ключи из **всех** ключей (классический выбор для чистого кэша).
* **`volatile-lru`** — вытесняются LRU-ключи, но только среди тех, у которых явно выставлен `TTL` (`EXPIRE`) — ключи без TTL считаются "постоянными" и не трогаются. Критично для смешанного использования (часть ключей — кэш, часть — постоянные данные вроде сессий с явным временем жизни).

> **DevOps-инсайт (сессии):** Хранение сессий пользователей в Redis вместо памяти конкретного backend-процесса — обязательный паттерн при горизонтальном масштабировании (несколько реплик приложения за Load Balancer). Без внешнего хранилища сессий балансировщик обязан использовать `IP Hash` (см. [Networking & Web Servers](../Networking-and-Web-Servers/README.md#web-servers)), чтобы конкретный пользователь всегда попадал на ту же ноду, где физически лежит его сессия в памяти процесса — что убивает гибкость масштабирования и Rolling Update (см. [CI/CD & GitOps](../CI-CD-and-GitOps/README.md#deployment-strategies)). Redis как внешнее хранилище сессий убирает эту привязку полностью — любая нода приложения может обслужить любого пользователя.

## Message Brokers

### 1. Зачем нужна асинхронная коммуникация между сервисами

Прямой синхронный HTTP-вызов между микросервисами (`Service A → HTTP → Service B`) означает, что A физически **блокируется**, ожидая ответа B, и падает целиком, если B недоступен. Message Broker разрывает эту жёсткую связь через промежуточную очередь:

```text
Service A → [публикует сообщение] → Broker → [хранит и доставляет] → Service B (в своём темпе)
```

* Если Service B временно упал или перегружен — сообщения **накапливаются** в брокере, а не теряются и не блокируют Service A.
* Producer (отправитель) и Consumer (получатель) полностью **развязаны** во времени, скорости обработки и даже технологическом стеке.

---

### 2. RabbitMQ: классический брокер на AMQP

**RabbitMQ** реализует протокол **AMQP (Advanced Message Queuing Protocol)** — модель с явной маршрутизацией через промежуточные **Exchange**.

#### Архитектура: Producer → Exchange → Queue → Consumer

```text
[Producer] → publish → [Exchange] → routing (по правилам) → [Queue 1] → [Consumer 1]
                                                        └──→ [Queue 2] → [Consumer 2]
```

Ключевое архитектурное решение AMQP: Producer **никогда** не публикует напрямую в очередь — он публикует в **Exchange**, а Exchange по своим правилам решает, в какие очереди разложить копию сообщения.

| Тип Exchange | Логика маршрутизации | Применение |
| :--- | :--- | :--- |
| **Direct** | Точное совпадение routing key | Отправить конкретное сообщение в конкретную очередь по имени |
| **Fanout** | Игнорирует routing key, рассылает **во все** привязанные очереди | Broadcast-события — "заказ создан" должны узнать сразу email-сервис, аналитика и склад |
| **Topic** | Совпадение по паттерну с wildcard (`order.*.created`) | Гибкая маршрутизация по иерархии событий |
| **Headers** | Маршрутизация по заголовкам сообщения, а не по routing key | Редкий случай, когда критерии маршрутизации сложнее одной строки |

#### Гарантии доставки: Acknowledgments

```text
Consumer получил сообщение → обрабатывает → ACK (подтверждение) → RabbitMQ удаляет сообщение из очереди
                                        ↘ если Consumer упал ДО ACK → сообщение возвращается в очередь (requeue)
```

* **Manual ACK (рекомендуется в проде):** Consumer явно подтверждает обработку **после** успешного завершения бизнес-логики. Если процесс упал посередине обработки — RabbitMQ видит разрыв соединения и возвращает сообщение другому consumer'у.
* **Auto ACK:** Сообщение считается доставленным сразу при получении, до реальной обработки — риск потери сообщения при падении consumer'а в процессе работы. Годится только для некритичных данных (метрики, необязательные уведомления).

#### Dead Letter Queue (DLQ) — сообщения, которые не удалось обработать

Если Consumer раз за разом падает на одном и том же "ядовитом" сообщении (например, из-за бага парсинга), без DLQ это сообщение будет **бесконечно** requeue'иться, забивая очередь и нагружая CPU повторными попытками.

```text
[Основная Queue] → N неудачных попыток обработки (retry с ограничением) → [Dead Letter Queue]
                                                                                  ↓
                                                          Инженер разбирает вручную / алертит мониторинг
```

DLQ настраивается через политику очереди (`x-dead-letter-exchange`) — после исчерпания лимита попыток сообщение автоматически перекладывается в отдельную очередь для ручного разбора, не блокируя обработку остальных нормальных сообщений.

### 3. Apache Kafka: распределённый event streaming

Kafka решает принципиально другую задачу, чем RabbitMQ — не "доставить сообщение и удалить", а **хранить непрерывный, воспроизводимый поток событий** для множества независимых потребителей.

#### Ключевое архитектурное отличие от классических очередей

| Критерий | RabbitMQ (Queue) | Kafka (Log) |
| :--- | :--- | :--- |
| **Модель хранения** | Сообщение удаляется из очереди сразу после успешного ACK. | Сообщения (events) хранятся в **логе** заданное время (`retention`, например 7 дней) **независимо** от того, прочитаны они или нет. |
| **Повторное чтение** | Невозможно — сообщение прочитано и удалено один раз. | Consumer может **перечитать** старые события с любой позиции (`offset`) — например, для пересчёта аналитики за прошлую неделю после исправления бага. |
| **Множественные consumer'ы одного сообщения** | Каждое сообщение обрабатывается **одним** consumer'ом (если не fanout в разные очереди). | Множество независимых **Consumer Groups** читают один и тот же поток **параллельно и независимо** — каждая группа хранит свой собственный offset. |
| **Порядок сообщений** | В рамках одной очереди — строгий FIFO. | Строгий порядок гарантирован только **внутри одной партиции**, не по всему топику. |
| **Типичное применение** | Task queues, RPC-подобные асинхронные вызовы, уведомления. | Event Sourcing, стриминг аналитики, агрегация логов между множеством разнородных систем, CDC (Change Data Capture). |

#### Топики и партиции

```text
Topic: "user-events"
├── Partition 0: [event0] [event3] [event6] [event9]  → offset растёт монотонно внутри партиции
├── Partition 1: [event1] [event4] [event7]
└── Partition 2: [event2] [event5] [event8]
```

* **Partition** — единица параллелизма и хранения в Kafka. Каждая партиция — упорядоченный, неизменяемый (append-only) лог.
* **Producer** решает, в какую партицию попадёт сообщение — обычно на основе хэша от `key` сообщения (например, `user_id`), что гарантирует: все события **одного** пользователя всегда попадают в одну и ту же партицию и, следовательно, сохраняют строгий порядок между собой.
* **Repication Factor** — каждая партиция реплицируется на несколько брокеров кластера (обычно 3), с одним **Leader**-брокером, принимающим запись, и остальными **Follower**-брокерами — концептуальный аналог Primary/Replica из [Relational Databases](#rdbms), только на уровне партиции, а не всей базы.

#### Consumer Groups: параллелизм чтения

```text
Topic "orders" (3 партиции)
        ↓
Consumer Group "analytics"          Consumer Group "email-service"
├── Consumer 1 → Partition 0        ├── Consumer 1 → Partition 0, 1, 2 (один consumer читает всё)
├── Consumer 2 → Partition 1
└── Consumer 3 → Partition 2
```

* Внутри **одной** Consumer Group каждая партиция читается **ровно одним** consumer'ом этой группы — это и даёт горизонтальное масштабирование обработки (добавил Consumer — Kafka перебалансировал партиции между ними, вплоть до одной партиции на consumer).
* **Разные** Consumer Group полностью независимы друг от друга — группа "analytics" и группа "email-service" читают один и тот же топик "orders" каждая в своём темпе, с собственным offset, не влияя друг на друга.

> **DevOps-инсайт (data pipelines):** Именно возможность **множественных независимых consumer'ов** и **воспроизводимости** потока делает Kafka стандартом для построения Data Pipeline: один и тот же поток событий "заказ создан" одновременно потребляют realtime-аналитика, ML-модель рекомендаций, аудит-лог и email-сервис — без необходимости producer'у знать о них заранее или дублировать публикацию в разные системы.

## Identity & Access

### 1. Keycloak: концепт Single Sign-On (SSO)

**Keycloak** — open-source сервер управления идентификацией (Identity and Access Management), решающий задачу: у компании 10 внутренних сервисов, и пользователь не должен логиниться в каждый отдельно.

```text
[Пользователь] → логинится ОДИН раз в Keycloak
        ↓
[Service A]  [Service B]  [Service C]  → каждый доверяет токену от Keycloak, не имея своей базы паролей
```

* **Realm** — изолированное пространство пользователей, ролей и клиентов в Keycloak (аналог отдельного "тенанта" — например, `internal-tools` realm отдельно от `customer-portal` realm).
* **Client** — конкретное приложение, зарегистрированное в realm, которому разрешено запрашивать аутентификацию через Keycloak.
* **Identity Provider (IdP) Federation** — Keycloak сам может выступать прослойкой к внешним провайдерам (Google, GitHub, корпоративный LDAP/Active Directory), давая пользователю "Войти через Google" даже если конечное приложение об этом ничего не знает — оно просто доверяет токену Keycloak.

---

### 2. OAuth2 vs OIDC: разграничение ответственности

Частая путаница на собеседованиях — эти протоколы решают **разные** задачи и работают вместе, а не заменяют друг друга.

| Протокол | Отвечает на вопрос | Что возвращает |
| :--- | :--- | :--- |
| **OAuth 2.0** | *«Какие права (scope) доступа выдать этому приложению?»* — протокол **авторизации**. | **Access Token** — билет с ограниченными правами на конкретные API-ресурсы. |
| **OIDC (OpenID Connect)** | *«Кто этот пользователь?»* — надстройка над OAuth2 для **аутентификации**. | **ID Token** (формат JWT) — подписанное удостоверение личности с полями (`sub`, `email`, `name`). |

> **Аналогия:** OAuth2 — это как выдать валетеру ключ от машины, ограниченный конкретными действиями (только парковка, без доступа в багажник) — авторизация без раскрытия личности владельца. OIDC поверх этого добавляет удостоверение личности — валетер ещё и точно знает, что перед ним хозяин машины, а не случайный человек.

#### Authorization Code Flow (типовой сценарий веб-приложения)

```text
1. Пользователь → жмёт "Войти" на Service A
2. Service A → редиректит на Keycloak (Authorization Endpoint)
3. Пользователь → вводит логин/пароль ПРЯМО на странице Keycloak (Service A пароль не видит вообще)
4. Keycloak → редиректит обратно на Service A с одноразовым Authorization Code
5. Service A (backend, не браузер) → обменивает Code на Access Token + ID Token через Token Endpoint
6. Service A → используя ID Token, узнаёт личность пользователя; используя Access Token — обращается к API от его имени
```

Критичная деталь шага 5: обмен **Code на токены** происходит на стороне backend'а Service A напрямую с Keycloak (server-to-server), а не через браузер пользователя — сами токены никогда не проходят через URL или JavaScript в браузере, что защищает их от перехвата.

#### JWT (JSON Web Token): формат, в котором путешествуют токены

```text
header.payload.signature
```

* **Header** — алгоритм подписи (например, `RS256`).
* **Payload** — сами данные (claims): `sub` (ID пользователя), `exp` (время истечения), `roles`.
* **Signature** — криптографическая подпись Keycloak, позволяющая **любому** сервису проверить подлинность токена **без похода в Keycloak за каждым запросом** — токен самодостаточен, сервис локально проверяет подпись публичным ключом Keycloak.

> **DevOps-инсайт (связь с CI/CD):** Тот же принцип самодостаточного подписанного токена лежит в основе OIDC-паттерна в GitHub Actions → AWS STS, разобранного в [CI/CD & GitOps](../CI-CD-and-GitOps/README.md#cicd-architecture) — GitHub выступает своим собственным мини-Keycloak, выпуская JWT, которому AWS доверяет по опубликованному публичному ключу, без единого постоянного пароля между системами.

#### Роли Keycloak в инфраструктуре кластера

На практике Keycloak чаще всего разворачивается как ArgoCD Application (тот же паттерн App-of-Apps, что и для ArgoCD/Prometheus/Loki в [`apps/applications`](../../apps/applications) этого репозитория) и встаёт перед Ingress Controller (см. [Containerization & Orchestration](../Containerization-and-Orchestration/README.md#ingress)) как единая точка аутентификации для всех внутренних веб-сервисов кластера — Grafana, ArgoCD UI и прочие внутренние дашборды могут делегировать логин ему вместо собственных механизмов паролей.
