## Модели OSI и TCP/IP

В реальной инженерии модель OSI используется как общий язык для локализации проблемы («проблема на L7» или «проверить L3/L4»). Модель TCP/IP — это то, как стек протоколов реально реализован в коде ядра Linux.

### Сводная шпаргалка по уровням

| Уровень OSI | Название уровня | Единица (PDU) | Протоколы / Технологии | Что делает DevOps на этом уровне? | Команды / Инструменты |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **L7** | Application (Прикладной) | Данные (Data) | HTTP, HTTPS, DNS, DHCP, SSH, gRPC | Настройка Nginx, Ingress, логика приложений, SSL-сертификаты. | `curl`, `dig`, `nc` |
| **L6** | Presentation (Представительский) | Данные (Data) | TLS/SSL, Сжатие, Кодировки | Настройка шифрования, TLS-терминация. | `openssl s_client` |
| **L5** | Session (Сеансовый) | Данные (Data) | RPC, Sockets, сессии | Поддержание соединений (Keep-Alive), тайм-ауты. | `ss -tulpn` |
| **L4** | Transport (Транспортный) | Сегмент (TCP) / Датаграмма (UDP) | **TCP, UDP** | Проброс портов (port-forwarding), балансировка нагрузки (L4 / NLB), firewall. | `nc`, `iptables`, `nftables` |
| **L3** | Network (Сетевой) | Пакет (Packet) | **IP**, ICMP, IPsec, Routing | Маршрутизация, подсети (CIDR), NAT, VPN, CNI-плагины в K8s. | `ping`, `ip route`, `traceroute` |
| **L2** | Data Link (Канальный) | Кадр (Frame) | Ethernet, ARP, VLAN | Настройка сетевых мостов (Bridge) в KVM/Docker, veth-пары. | `ip link`, `ip neigh`, `bridge` |
| **L1** | Physical (Физический) | Бит (Bit) | Кабели, патч-панели, Wi-Fi | Проверка линка на сетевой карте в ЦОД. | `ethtool` |



### Главный DevOps-инсайт (Инкапсуляция)

Когда приложение отправляет HTTP-запрос (L7), ядро Linux оборачивает его в TCP-заголовок (L4), затем в IP-заголовок (L3), затем в Ethernet-кадр (L2) и отправляет в интерфейс (L1). 

При траблшутинге мы всегда спускаемся сверху вниз по цепочке:
> Домен резолвится? (L7) $\rightarrow$ Хост пингуется? (L3) $\rightarrow$ Порт открыт? (L4) $\rightarrow$ TLS-сертификат валиден? (L6) $\rightarrow$ HTTP-код ответа 200? (L7).

## Транспортный уровень (L4): TCP vs UDP

### TCP (Transmission Control Protocol)
Протокол с установлением соединения (Connection-oriented) и гарантией доставки.

* **TCP 3-Way Handshake (Трехэтапное рукопожатие):**
  1. Клиент $\rightarrow$ `SYN` *(Хочу подключиться, мой начальный Sequence Number = X)*
  2. Сервер $\rightarrow$ `SYN-ACK` *(Слышу тебя, подтверждаю X, мой начальный Sequence Number = Y)*
  3. Клиент $\rightarrow$ `ACK` *(Подтверждаю Y, устанавливаю связь)*

* **Особенности:** Гарантирует порядок пакетов, контролирует перегрузку сети (Congestion Control: BBR, CUBIC), переотправляет потерянные пакеты.
* **Применение в DevOps:** Базы данных (PostgreSQL, MySQL), SSH, HTTP/HTTPS (до версии HTTP/3).

### UDP (User Datagram Protocol)
Протокол без установления соединения (Connectionless) и без гарантии доставки.

* **Как работает:** Отправляет дейтаграммы без подтверждения приема (Fire and Forget). Если пакет потерялся — UDP об этом не узнает.
* **Особенности:** Нет оверхеда на установление связи, минимальные задержки. Контроль целостности и порядка (если он необходим) переносится на уровень самого приложения.
* **Применение в DevOps:** DNS, DHCP, StatsD (метрики), потоковый syslog, WireGuard, HTTP/3 (QUIC).

### Важные состояния TCP-соединения (Инструмент для интервью)

При анализе сети на сервере состояния сокетов скажут о многом:

* `LISTEN` — Сервер ждет входящих подключений на порту (например, Nginx слушает `0.0.0.0:80`).
* `ESTABLISHED` — Соединение успешно установлено, идет активный обмен данными.
* `TIME_WAIT` — Соединение закрыто со стороны клиента. ОС держит сокет открытым еще некоторое время (обычно $2 \times \text{MSL} = 60-120$ секунд), чтобы поймать запоздавшие пакеты.
  * *Проблема:* Если на высоконагруженном прокси-сервере плодится слишком много `TIME_WAIT`, могут закончиться свободные локальные (эфемерильные) порты для новых коннектов к бэкенду. 
  * *Решение:* Тюнинг ядра через `sysctl`:
    ```
    net.ipv4.tcp_tw_reuse = 1
    ```
* `CLOSE_WAIT` — Удаленная сторона (клиент) закрыла соединение, но наше приложение (бэкенд) еще не вызвало `close()` для сокета со своей стороны.
  * *Проблема:* Если в выводе утилит висит много `CLOSE_WAIT` — это **100% баг в коде приложения** (утечка файловых дескрипторов). Тюнинг ядра тут не поможет, нужно пинать разработчиков.

## Сетевая инфраструктура (L2-L3): NAT, Bridge, DHCP

### Сетевой мост (Bridge)
Виртуальный или программный коммутатор уровня L2.

* **Как работает:** Объединяет несколько сетевых интерфейсов в одну локальную сеть. Пакеты пересылаются на основе таблицы MAC-адресов.
* **В виртуализации (Docker/KVM):** При старте Docker на хосте создается виртуальный интерфейс `docker0` (тип bridge). Каждый запущенный контейнер получает свою виртуальную пару интерфейсов `veth` (один конец внутри контейнера как `eth0`, второй — «вткнут» в мост `docker0`). Все контейнеры в дефолтной сети находятся в одном L2-домене и видят друг друга напрямую без маршрутизации.

### NAT (Network Address Translation)
Трансляция сетевых адресов на уровне L3.



* **Зачем нужен:** Позволяет экономить публичные IPv4-адреса и скрывать внутреннюю структуру сети от внешнего мира. Роутер переписывает IP-адреса в заголовках пакетов «на лету».
* **SNAT (Source NAT / Masquerading):** Подменяет внутренний приватный IP-адрес хоста на публичный IP-адрес роутера при выходе пакета во внешнюю сеть. 
  * *Зачем DevOps:* Позволяет виртуалкам в приватных подсетях (например, AWS VPC Private Subnet через NAT Gateway) или контейнерам Docker безопасно ходить в интернет за обновлениями и пакетами.
* **DNAT (Destination NAT / Port Forwarding):** Подменяет публичный IP-адрес и порт назначения на приватный IP и порт внутри сети. 
  * *Зачем DevOps:* Когда ты запускаешь контейнер командой с пробросом портов, Docker под капотом вешает правило DNAT в `iptables`. Трафик с внешнего порта хоста перенаправляется на внутренний IP контейнера:
    
    ```
    docker run -p 8080:80 nginx
    ```

## Безопасность и шифрование: SSH, TLS/SSL, VPN

### SSH (Secure Shell)
Протокол удаленного управления на уровне L7 с шифрованием трафика.

* **Конфигурация:** Основной файл настроек демона — `/etc/ssh/sshd_config`.
* **DevOps Best Practices (для интервью и проде):**
  * Отключить вход по паролям, оставить только ключи: `PasswordAuthentication no`.
  * Отключить прямой доступ для пользователя root: `PermitRootLogin no`.
  * Сменить дефолтный порт `22` на любой свободный нетипичный порт для защиты от автоматических брутфорс-ботов.
  * Использовать современные криптостойкие ключи (например, `ED25519` вместо устаревших `RSA`).

### TLS/SSL Handshake (Шифрование на L6)
Обеспечивает безопасную передачу данных по HTTPS. Процесс установления сессии выглядит так:

1. **Client Hello:** Клиент отправляет серверу поддерживаемые им версии TLS и список алгоритмов шифрования (Cipher Suites).
2. **Server Hello + Certificate:** Сервер выбирает наиболее безопасный общий шифр и отправляет клиенту свой публичный SSL-сертификат.
3. **Проверка и обмен ключами:** Клиент проверяет валидность сертификата через цепочку доверенных центров сертификации (CA). Затем клиент и сервер с помощью асимметричного шифрования (RSA или Diffie-Hellman) генерируют общий **симметричный сессионный ключ**.
4. **Зашифрованная сессия:** Асимметричное шифрование завершается (оно слишком тяжелое для CPU). Все последующие данные внутри HTTPS-сессии шифруются быстро со смешанным симметричным ключом.

### VPN: WireGuard vs OpenVPN

* **OpenVPN:**
  * *Архитектура:* Классический тяжеловесный VPN на базе OpenSSL. Может работать как на L3 (`tun`), так и на L2 (`tap`).
  * *Минусы:* Огромная кодовая база, сложная настройка, работает в user-space (из-за чего тратит ресурсы на контекстные переключения между ядром и пространством пользователя), долго восстанавливает связь при смене сети.
* **WireGuard:**
  * *Архитектура:* Современный легковесный протокол, интегрированный **напрямую в ядро Linux**. Работает строго на L3 поверх UDP.
  * *Плюсы:* Очень высокая пропускная способность и минимальный пинг. Кодовая база составляет всего ~4000 строк (легко аудировать). Не держит постоянное соединение: если трафика нет — молчит, при смене сети (например, переключении Wi-Fi/LTE) восстанавливает связь мгновенно. Использует фиксированный набор современной криптографии (ChaCha20, Poly1305).

## Уровень приложений (L7): Протокол HTTP/HTTPS

### Структура HTTP-обмена
Каждое взаимодействие в HTTP состоит из двух частей:
* **Запрос (Request):** Метод (`GET`, `POST`, `PUT`, `DELETE`), URI (путь к ресурсу), заголовки и опциональное тело (например, JSON-пейлод).
* **Ответ (Response):** Статус-код, заголовки и тело ответа (HTML, JSON, изображение).

---

### Важные HTTP-заголовки (Headers)

Заголовки позволяют передавать метаданные между клиентом и сервером. Для DevOps критичны следующие:

* `Host` — Доменное имя сервера, к которому обращается клиент. Именно по этому заголовку Nginx понимает, в какой блок `server { server_name ... }` направить трафик, если на одном IP-адресе висит несколько сайтов.
* `Authorization` — Передача учетных данных для проверки подлинности. Чаще всего используется схема `Bearer <JWT_TOKEN>` или `Basic <base64-строка>`.
* `Content-Type` — Указывает формат данных в теле запроса или ответа (например, `application/json`, `text/html`, `multipart/form-data`).
* `X-Forwarded-For` / `X-Real-IP` — Проставляются прокси-сервером (Nginx/HAProxy). Позволяют передать реальный IP-адрес пользователя на бэкенд. Без них бэкенд будет думать, что все запросы идут локально от самого Nginx.
* `X-Forwarded-Proto` — Передает исходный протокол (`http` или `https`). Помогает бэкенду понять, что TLS-терминация произошла на балансировщике и трафик изначально был защищен.

### Классификация статус-кодов (Инструмент для траблшутинга)



#### 2xx (Success) — Успешно
* `200 OK` — Стандартный успешный запрос.
* `201 Created` — Запрос выполнен успешно и привел к созданию нового ресурса (частый ответ на `POST`/`PUT`).
* `204 No Content` — Запрос обработан успешно, но сервер намеренно не возвращает тело ответа (часто при `DELETE`).

#### 3xx (Redirect) — Перенаправление
* `301 Moved Permanently` — Ресурс навсегда переехал на новый URI. Браузеры кэшируют этот ответ намертво. Используется для жесткого редиректа с HTTP на HTTPS.
* `302 Found` (или `307 Temporary Redirect`) — Временный редирект. Запрос должен быть повторен по новому адресу, но кэшировать его нельзя.
* `304 Not Modified` — Контент не менялся со времени последнего запроса. Сервер не гонит данные заново, а говорит браузеру взять их из локального кэша.

#### 4xx (Client Error) — Ошибка клиента
* `400 Bad Request` — Сервер не смог распарсить запрос (битый JSON, синтаксическая ошибка в параметрах).
* `401 Unauthorized` — Клиент не аутентифицирован. Не передан или просрочен токен авторизации.
* `403 Forbidden` — Аутентификация пройдена, но у текущего пользователя банально нет прав/ролей для доступа к этому эндпоинту.
* `404 Not Found` — Запрашиваемый URI не существует на сервере.
* `429 Too Many Requests` — Клиент превысил лимит запросов в единицу времени. Сработал механизм Rate Limiting.

#### 5xx (Server Error) — Ошибка сервера
* `500 Internal Server Error` — Любая необработанная ошибка в коде самого бэкенд-приложения (unhandled exception). Сервер упал на операции.
* `502 Bad Gateway` — Nginx принял запрос, но бэкенд, стоящий за ним (Node.js-процесс, PHP-FPM, Go-бинарник), мертв, упал или вообще не слушает указанный сокет/порт.
* `503 Service Unavailable` — Сервер временно не может обрабатывать запросы (перегрузка, превышение пула соединений к БД или плановое обслуживание).
* `504 Gateway Timeout` — Nginx успешно перенаправил запрос на бэкенд, но тот завис (например, на тяжелом SQL-запросе) и не успел отдать ответ в рамках таймаута:
  ```
  proxy_read_timeout 60s;
  ```

## Веб-серверы: Nginx vs Apache

### Архитектурные различия

| Критерий | Nginx | Apache |
| :--- | :--- | :--- |
| **Архитектура** | **Event-driven (Асинхронная)**. Один рабочий процесс (worker) на ядро CPU. Обрабатывает тысячи соединений одновременно в одном потоке за счет неблокирующего ввода-вывода (`epoll` в Linux). | **Process/Thread-driven**. Выделяет отдельный поток или процесс на каждое новое входящее соединение (модули `mpm_prefork`, `worker` или `event`). |
| **Потребление RAM** | Низкое и стабильное. Потребление памяти практически не растет с ростом количества открытых соединений. | Высокое. Растет линейно. При резком наплыве пользователей (C10k проблема) сервер может легко уйти в Swap или упасть по OOM. |
| **Раздача статики** | Молниеносная. Отдает файлы напрямую с диска в сетевой сокет в обход user-space с помощью системного вызова `sendfile`. | Медленнее. Тратит на обработку простых файлов полноценные тяжелые процессы/потоки. |
| **Динамический контент** | Не умеет обрабатывать динамику (код) внутри себя. Перенаправляет запросы на внешние менеджеры процессов (PHP-FPM, Node.js через сокеты/порты). | Может обрабатывать динамику прямо внутри своего процесса с помощью встроенных модулей (например, `mod_php`, `mod_perl`). |
| **Конфигурация** | Только централизованная (изменения вносятся администратором в файлы `/etc/nginx/`). Требует перезагрузки (`reload`) для применения. | Поддерживает децентрализованную настройку через файлы `.htaccess` в папках сайтов. Удобно для shared-хостингов, но сильно замедляет работу (Apache вынужден перечитывать файлы при каждом запросе). |

### Ключевые концепты Nginx



#### 1. Reverse Proxy (Обратный прокси-сервер)
Nginx принимает запросы из внешнего мира (клиенты) и перенаправляет их на внутренние серверы приложений (бэкенды), которые скрыты в приватной сети. 
* **Зачем:** Безопасность (бэкенд не смотрит в интернет), логирование, кэширование статики, защита от DDoS и **TLS-терминация** (Nginx берет на себя тяжелую математику по шифрованию HTTPS, а с бэкендом общается по простому HTTP внутри локалки).

#### 2. Балансировка нагрузки (Load Balancing)
Nginx умеет распределять входящие запросы между несколькими экземплярами (нодами) приложения для обеспечения отказоустойчивости и горизонтального масштабирования.

**Основные алгоритмы балансировки:**
* `Round Robin` (по умолчанию) — Запросы распределяются по очереди строго один за другим.
* `Weighted Round Robin` — Запросам на более мощные серверы задается вес (`weight`). Нода с весом 3 получит в три раза больше запросов, чем нода с весом 1.
* `Least Connections` (`least_conn`) — Запрос отправляется на сервер с наименьшим количеством активных соединений на данный момент. Отлично подходит для "тяжелых" запросов.
* `IP Hash` (`ip_hash`) — Сервер выбирается на основе IP-адреса клиента. Гарантирует, что конкретный пользователь всегда будет попадать на одну и ту же ноду бэкенда (важно, если сессии хранятся локально в памяти приложения, а не в Redis).

### Шаблон production-конфига Nginx

Ниже представлен базовый пример конфигурации виртуального хоста с балансировкой, TLS-терминацией и разделением статики/динамики:

```
# Блок балансировщика
upstream my_app_backend {
    zone backend_connections 64k;
    server 127.0.0.1:3000 weight=3;
    server 127.0.0.1:3001;
    keepalive 32; # Поддержка постоянных соединений с бэкендом
}

# Редирект с HTTP на HTTPS
server {
    listen 80;
    server_name app.dante.local;
    return 301 https://$host$request_uri;
}

# Основной HTTPS сервер
server {
    listen 443 ssl;
    server_name app.dante.local;

    # Настройки SSL
    ssl_certificate /etc/ssl/certs/app.crt;
    ssl_certificate_key /etc/ssl/private/app.key;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;

    # Раздача статики напрямую (Молниеносно)
    location /static/ {
        root /var/www/my_app;
        expires 30d;
        add_header Cache-Control "public, no-transform";
    }

    # Проксирование всего остального на бэкенд
    location / {
        proxy_pass http://my_app_backend;
        
        # Настройка Keep-Alive с бэкендом
        proxy_http_version 1.1;
        proxy_set_header Connection "";

        # Передача реальных заголовков клиента
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # Тайм-ауты
        proxy_connect_timeout 5s;
        proxy_read_timeout 60s;
    }
}
```


<a id="troubleshooting"></a>

## Практический Troubleshooting-бокс (Инструментарий)

Сценарий траблшутинга всегда идет **снизу вверх** по модели OSI (от L3 к L7):
> Проверяем физический линк / маршрут (L3) $\rightarrow$ Проверяем доступность портов (L4) $\rightarrow$ Дебажим прикладной уровень, TLS и DNS (L6/L7).

---

### 1. Сетевая связность и маршрутизация (L3)

#### ping
Проверяет базовую доступность узла по протоколу ICMP.
* *DevOps-инсайт:* Если `ping` выдает таймаут, это не значит, что сервис лежит. В облаках (например, в AWS Security Groups) ICMP-трафик часто заблокирован по умолчанию из соображений безопасности, при этом сам веб-сайт (порт 443) может работать отлично.

#### traceroute / mtr
Показывает путь пакета через промежуточные маршрутизаторы (хопы).
* `mtr` (My Traceroute) комбинирует `traceroute` и `ping`, обновляя статистику потерь пакетов на каждом хопе в реальном времени. Помогает понять, на чьей стороне проблемы — твоего провайдера, магистрального оператора или дата-центра.

---

### 2. Сокеты и локальные порты (L4)

#### ss (замена устаревшего netstat)
Показывает состояние сетевых сокетов в ядре Linux.

* **Найти, какой процесс занял порт и мешает запуску сервиса:**
  ```
  sudo ss -tulpn | grep :80
  # -t (tcp), -u (udp), -l (listening сокеты), -p (показать PID и имя процесса), -n (показывать порты цифрами)
  ```

* **Посчитать количество соединений в разных состояниях (оценка нагрузки):**
  ```
  ss -tao | awk '{print $1}' | sort | uniq -c
  # Позволяет быстро увидеть всплеск TIME_WAIT или CLOSE_WAIT
  ```

#### nc (netcat)
«Швейцарский нож» системного администратора. Используется для быстрой проверки доступности удаленных портов.

* **Проверить, открыт ли порт на удаленном сервере (вместо тяжелого telnet):**
  ```
  nc -zv 192.168.1.50 443
  # -z (режим zero-I/O, сканирование), -v (verbose)
  ```

---

### 3. Диагностика DNS (L7)

#### dig (замена устаревшего nslookup)
Инструмент для ручных DNS-запросов и дебага DNS-зон.

* **Получить краткий и чистый ответ (только IP, идеально для скриптов автоматизации):**
  ```
  dig example.com +short
  ```

* **Запросить конкретный тип записи напрямую у публичного сервера (минуя локальный кэш):**
  ```
  dig @8.8.8.8 example.com TXT
  # Показывает, обновилась ли SPF/TXT запись в мире
  ```

* **Трейсинг DNS-резолвинга от корня (как DNS-сервер ищет имя):**
  ```
  dig example.com +trace
  ```

---

### 4. Веб-дебаг и HTTP/TLS (L6/L7)

#### curl
Утилита для взаимодействия с серверами по HTTP/HTTPS/FTP.

* **Посмотреть заголовки ответа, этапы TLS-handshake и параметры сессии:**
  ```
  curl -v https://example.com
  # -v (verbose) показывает IP, детали SSL-сертификата и заголовки запроса (>) и ответа (<)
  ```

* **Забрать только заголовки ответа (проверка редиректов и кэш-заголовков):**
  ```
  curl -I https://example.com
  ```

* **Сделать запрос, игнорируя проверку самоподписанного SSL-сертификата (полезно для staging-окружений):**
  ```
  curl -k https://dev.local
  # -k (--insecure)
  ```

---

### 5. Анализ трафика "на проводе" (Глубокий дебаг L2-L4)

#### tcpdump
Консольный сниффер сетевых пакетов. Позволяет увидеть, доходят ли пакеты до сетевой карты вообще. Требует прав `root`.

* **Перехват трафика на определенном порту и интерфейсе:**
  ```
  sudo tcpdump -i eth0 port 80 -nn
  # -i (интерфейс), -nn (не резолвить IP в имена хостов и порты в имена сервисов — ускоряет работу)
  ```

* **Мониторинг трафика DHCP (DORA) в реальном времени:**
  ```
  sudo tcpdump -lnni any port 67 or port 68
  ```

* **Сохранить дамп пакетов в файл для последующего анализа в GUI Wireshark:**
  ```
  sudo tcpdump -i any host 1.2.3.4 -w /tmp/capture.pcap
  ```

---

### 6. Парсинг логов Nginx (Скрипты на коленке для интервью)

На собеседованиях часто дают задачу быстро проанализировать текстовый лог без использования систем мониторинга.

* **Топ-10 IP-адресов, которые делают больше всего запросов (поиск DDoS/парсеров):**
  ```
  awk '{print $1}' /var/log/nginx/access.log | sort | uniq -c | sort -nr | head -n 10
  # Считываем первый столбец (IP), сортируем для группировки, считаем дубли, сортируем по убыванию чисел, берем топ-10
  ```

* **Мониторинг 5xx ошибок в логе в реальном времени:**
  ```
  tail -f /var/log/nginx/access.log | grep --line-buffered '" 50'
  ```
