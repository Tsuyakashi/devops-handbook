## Модели OSI и TCP/IP

В реальной инженерии модель OSI используется как общий язык для локализации проблемы («проблема на L7» или «проверить L3/L4»). Модель TCP/IP — это то, как стек протоколов реализован в коде ядра Linux.

### Сводная шпаргалка по уровням

| Уровень OSI | Название уровня | Единица данных (PDU) | Протоколы / Технологии | Что делает DevOps на этом уровне? | Команды / Инструменты |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **L7** | Application (Прикладной) | Данные (Data) | HTTP, HTTPS, DNS, DHCP, SSH, FTP, gRPC | Настройка Nginx, Ingress, логика приложений, SSL-сертификаты. | `curl`, `dig`, `nc` |
| **L6** | Presentation (Представительский) | Данные (Data) | TLS/SSL, Сжатие, Кодировки | Настройка шифрования, TLS-терминация. | `openssl s_client` |
| **L5** | Session (Сеансовый) | Данные (Data) | RPC, Sockets, сессии | Поддержание соединений (Keep-Alive), тайм-ауты. | `ss -tulpn` |
| **L4** | Transport (Транспортный) | Сегмент (TCP) / Дейтаграмма (UDP) | **TCP, UDP** | Проброс портов (port-forwarding), балансировка нагрузки (L4 Балансировщики), firewall. | `telnet`, `nc`, `iptables` |
| **L3** | Network (Сетевой) | Пакет (Packet) | **IP**, ICMP, IPsec | Маршрутизация, подсети (CIDR), NAT, VPN, маршруты между подами в K8s. | `ping`, `ip route`, `traceroute` |
| **L2** | Data Link (Канальный) | Кадр (Frame) | Ethernet, ARP, VLAN | Настройка сетевых мостов (Bridge) в KVM/Docker, проверка MAC-адресов. | `ip link`, `arp -an` |
| **L1** | Physical (Физический) | Бит (Bit) | Кабели, патч-панели, Wi-Fi | Проверка линка на сетевой карте дата-центра. | `ethtool` |

### Главный DevOps-инсайт (Инкапсуляция)
Когда приложение отправляет HTTP-запрос (L7), ядро Linux оборачивает его в TCP-заголовок (L4), затем в IP-заголовок (L3), затем в Ethernet-кадр (L2) и отправляет в провод (L1). При траблшутинге мы всегда спускаемся сверху вниз: «Домен резолвится? (L7) -> Хост пингуется? (L3) -> Порт открыт? (L4) -> HTTP-код 200? (L7)».

## Transport Layer: TCP vs UDP

### TCP (Transmission Control Protocol)
Протокол с установлением соединения (Connection-oriented) и гарантией доставки.
* **Как работает:** Перед передачей данных происходит **TCP 3-Way Handshake (Трехэтапное рукопожатие)**:
  1. Клиент -> `SYN` (Хочу подключиться)
  2. Сервер -> `SYN-ACK` (Слышу тебя, готов)
  3. Клиент -> `ACK` (Понял, устанавливаю связь)
* **Плюсы:** Гарантирует порядок пакетов, контролирует перегрузку сети (Congestion Control), переотправляет потерянные пакеты.
* **Где нужен в DevOps:** Базы данных (PostgreSQL, MySQL), SSH, HTTP/HTTPS, API.

### UDP (User Datagram Protocol)
Протокол без установления соединения (Connectionless) и без гарантии доставки.
* **Как работает:** Пакеты отправляются «в никуда» (Fire and Forget). Если пакет потерялся по дороге — UDP об этом не узнает.
* **Плюсы:** Минимальные задержки (нет оверхеда на установление связи и подтверждения).
* **Где нужен в DevOps:** DNS (быстрые запросы), DHCP, Системы логирования/метрик (StatsD, потоковый syslog), VPN (WireGuard работает поверх UDP), Стриминг/VoIP.

### Важные состояния TCP-соединения (для вывода `ss` или `netstat`)
* `LISTEN` — Сервер ждет входящих подключений на порту (например, Nginx слушает 80).
* `ESTABLISHED` — Соединение успешно установлено, идет обмен данными.
* `TIME_WAIT` — Соединение закрыто со стороны клиента, но ОС держит сокет открытым еще некоторое время, чтобы поймать «запоздавшие» пакеты. *Важно:* Если на сервере слишком много `TIME_WAIT`, могут кончиться свободные локальные порты.
* `CLOSE_WAIT` — Удаленная сторона закрыла соединение, а наше приложение еще не закрыло сокет со своей стороны (потенциальная утечка дескрипторов в коде).

## DNS (Domain Name System)
Служба перевода человеческих имен сайтов в IP-адреса. Работает преимущественно по UDP (порт 53), но при больших ответах (более 512 байт) или трансфере зон переключается на TCP.

### Основные типы DNS-записей (Core Records)

1. **A-запись (Address)**
   * **Что делает:** Связывает имя домена с IPv4-адресом.
   * **Пример:** `app.example.com.  IN  A  192.168.1.50`
2. **CNAME-запись (Canonical Name)**
   * **Что делает:** Алиас (ссылка) одного доменного имени на другое.
   * **Пример:** `www.example.com.  IN  CNAME  example.com.`
   * *DevOps-нюанс:* CNAME нельзя вешать на корень домена (Apex domain, т.е. просто `example.com`), только на поддомены. Для корня в облаках используют специфичные типы вроде ALIAS/ANAME.
3. **MX-запись (Mail Exchange)**
   * **Что делает:** Указывает на почтовые серверы, отвечающие за прием почты для этого домена. Имеет приоритет (чем меньше цифра, тем выше приоритет).
   * **Пример:** `example.com.  IN  MX  10  mail.example.com.`
4. **TXT-запись (Text)**
   * **Что делает:** Хранит любую текстовую информацию.
   * **Зачем DevOps:** Чаще всего используется для верификации прав на домен (в Google/Yandex Webmaster), а также для настройки безопасности почты:
     * **SPF** (каким IP разрешено отправлять почту от имени домена).
     * **DKIM** (публичный ключ для проверки подписи писем).
     * **DMARC** (правило, что делать с письмами, не прошедшими SPF/DKIM).

### Иерархия DNS и резолвинг (Кратко)
Когда мы запрашиваем `test.demo.com`:
1. Проверяется локальный `/etc/hosts` и кэш.
2. Запрос идет к **Local Resolver** (например, `8.8.8.8` или DNS провайдера).
3. Если у резолвера нет кэша, он идет к **Root hints** (`.`) -> получает IP серверов зоны `.com` -> идет туда -> получает IP серверов `demo.com` (**Authoritative DNS**) -> забирает финальный IP.

## DHCP (Dynamic Host Configuration Protocol)
Протокол автоматического назначения сетевых настроек хостам (IP, маска подсети, шлюз по умолчанию, DNS-серверы). Работает на L7 поверх UDP (порты 67 для сервера, 68 для клиента).

### Как работает: Алгоритм DORA

При подключении нового хоста к сети (или запуске пустой виртуалки) происходит 4 шага:

1. **D**iscover (Поиск) — Клиент отправляет широковещательный запрос (Broadcast, `255.255.255.255`) на всю сеть: *«Есть тут DHCP-сервер? Мне нужен IP»*.
2. **O**ffer (Предложение) — DHCP-сервер ловит запрос и предлагает свободный IP-адрес: *«Привет, у меня есть свободный IP 192.168.1.100, возьмешь?»*.
3. **R**equest (Запрос) — Клиент отвечает (тоже Broadcast, чтобы другие DHCP-серверы, если они есть, услышали): *«Да, отличный IP, я бронирую адрес 192.168.1.100 с сервера X»*.
4. **A**cknowledge (Подтверждение) — Сервер подтверждает аренду (Lease Time) и передает остальные параметры (шлюз, DNS): *«Ок, зафиксировал. Этот IP твой на 24 часа. Твой шлюз: 192.168.1.1. DNS: 8.8.8.8»*.

### DevOps-контекст: DHCP в автоматизации
* В облаках и виртуализации (KVM, AWS, OpenStack) DHCP-сервер встроен в виртуальный свитч/сеть. Когда `vagrant up` поднимает виртуалку с типом сети `dhcp`, именно по этому протоколу она бесшовно получает IP.
* **IP-MAC Binding (Static Lease):** В проде для критичных серверов (БД, балансировщики) IP прописывают статикой в конфигурационных файлах ОС, либо настраивают на DHCP-сервере жесткую привязку IP к MAC-адресу, чтобы сервер всегда получал один и тот же адрес.

## Базовый Troubleshooting-бокс (Что запустить, если...)

* **Проверить, открыт ли TCP-порт на удаленном сервере:**
  ```bash
  nc -zv 192.168.1.50 443
  # Или по-старинке (если нет nc):
  telnet 192.168.1.50 443

* **Посмотреть, какие порты слушают процессы прямо сейчас на машине:**

```Bash
ss -tulpn
# -t (tcp), -u (udp), -l (listening), -p (process name), -n (numeric ports)
Запросить конкретную DNS-запись у определенного DNS-сервера:
```

```Bash
dig @8.8.8.8 example.com TXT
# Краткий ответ (только IP):
dig example.com +short
```

* **Посмотреть трафик DHCP в реальном времени (сетевой дамп):**

```Bash
tcpdump -lnni any port 67 or port 68
```
