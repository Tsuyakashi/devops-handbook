## Configuration Management (Ansible)

### 1. Архитектура и базовые концепции Ansible

**Ansible** — это утилита для управления конфигурациями и деплоя, работающая по **Agentless** (безагентной) схеме. В отличие от Chef или Puppet, тебе не нужно устанавливать и обновлять тяжелые фоновые процессы (демоны) на целевых серверах.



* **Control Node (Управляющий узел):** Машина, на которой установлен сам Ansible. Именно здесь хранятся плейбуки и запускаются команды. Управляющим узлом может быть ноутбук инженера или воркер в CI/CD пайплайне.
* **Managed Nodes (Управляемые узлы):** Целевые сервера (хосты), на которые накатывается конфигурация. Единственные системные требования к ним — наличие SSH-сервера и установленный интерпретатор Python (через него Ansible выполняет свои модули).
* **Inventory (Инвентарь):** Файл конфигурации (в формате `INI` или `YAML`), описывающий список целевых хостов, разбитых по группам (например, `[webservers]`, `[databases]`), а также специфичные для них переменные.

---

### 2. Принцип работы: Идемпотентность и декларативность

Два главных столпа, на которых держится Ansible:

#### Идемпотентность (Idempotency)
Это свойство системы при повторном применении одной и той же операции выдавать один и тот же результат без изменения состояния самой системы. 
* *Пример:* Если в таске указано «установить пакет `nginx`», то при первом запуске Ansible установит пакет (`changed`). При повторном запуске он проверит систему, увидит, что `nginx` уже нужной версии, и просто пропустит этот шаг (`ok`), не тратя ресурсы и ничего не ломая.

#### Декларативный подход через императивные модули
Ты описываешь *желаемый конечный статус* ресурса (декларативность), а Ansible сам знает, какие Linux-команды нужно выполнить под капотом (императивность). 

#### Схема выполнения таски под капотом:
1. Control Node берет код модуля (например, `apt` или `template`) и генерирует из него временный Python-скрипт, упаковывая туда переданные тобой параметры.
2. Этот скрипт копируется на Managed Node через SSH в директорию `~/.ansible/tmp/`.
3. Скрипт запускается локально на целевом сервере, считывает текущее состояние системы, вносит изменения (если они требуются), собирает JSON-ответ и возвращает его на Control Node.
4. Временный скрипт удаляется с целевого хоста.

### 3. Структура Playbooks и лучшие практики

**Playbook (Плейбук)** — это конфигурационный файл на языке YAML, который определяет последовательность действий (Plays) для конкретных групп хостов из инвентаря.

#### Ключевые абстракции плейбука:
* **Play:** Связывает группу хостов (`hosts: webservers`) с набором задач и настроек (права, переменные).
* **Task (Задача):** Именованный блок, вызывающий конкретный модуль Ansible с нужными аргументами. Задачи выполняются строго сверху вниз, по очереди.
* **Handler (Обработчик):** Особый тип таски, который срабатывает *только* в том случае, если какая-то из основных задач вернула статус `changed`. Используется для перезапуска сервисов при изменении конфигов.

#### Пример базового плейбука для бэкенда Fastify:
```yaml
---
- name: Configure Fastify Application Servers
  hosts: webservers
  become: true # Выполнять команды от имени root (sudo)

  vars:
    app_dir: /var/www/agentic-news

  tasks:
    - name: Ensure Node.js and NodeJS repository are present
      ansible.builtin.apt:
        name: nodejs
        state: present
        update_cache: true

    - name: Copy Nginx configuration template
      ansible.builtin.template:
        src: templates/nginx.conf.j2
        dest: /etc/nginx/sites-available/fastify.conf
        mode: '0644'
      notify: Restart Nginx # Триггер для хэндлера

  handlers:
    - name: Restart Nginx
      ansible.builtin.service:
        name: nginx
        state: restarted
```

### 4. Масштабирование через Roles и экосистему Ansible Galaxy

Когда плейбук разрастается до тысяч строк, поддерживать его в одном файле становится невозможно. Для декомпозиции кода используется концепция **Ролей (Roles)**.

#### Стандартная структура директории роли (Ansible Role Layout):
Роль — это строго регламентированная структура папок, где файлы автоматически подтягиваются по их назначению:
```text
my_role/
├── tasks/
│   └── main.yml     # Главный список задач роли
├── handlers/
│   └── main.yml     # Обработчики (например, рестарты сервисов)
├── templates/
│   └── app.conf.j2  # Шаблоны конфигураций (Jinja2)
├── files/
│   └── script.sh    # Статичные файлы для копирования "как есть"
├── vars/
│   └── main.yml     # Переменные с высоким приоритетом (не для переопределения)
├── defaults/
│   └── main.yml     # Дефолтные переменные с низким приоритетом (легко переопределить)
└── meta/
    └── main.yml     # Метаданные роли (автор, зависимости от других ролей)
```

#### Ansible Galaxy
Это публичный репозиторий (хаб) готовых, написанных сообществом ролей для типовых задач (установка Docker, настройка PostgreSQL кластера, тюнинг firewall). 

* Чтобы не изобретать велосипед, ты описываешь зависимости в файле `requirements.yml`:
  ```yaml
  ---
  roles:
    - name: geerlingguy.docker
      version: 3.1.2
  ```
* И скачиваешь их одной командой на Control Node: `ansible-galaxy install -r requirements.yml`. Это позволяет держать свой собственный репозиторий чистым, переиспользуя проверенный годами enterprise-код.

## Infrastructure Provisioning (Terraform / OpenTofu)

### 1. Ядро Terraform и OpenTofu: Декларативный IaC

**Terraform** (и его полностью открытый форк **OpenTofu**) — это инструменты для оркестровки инфраструктуры, которые позволяют описывать облачные и физические ресурсы с помощью человекочитаемого декларативного языка **HCL (HashiCorp Configuration Language)**.



* **Императивный vs Декларативный IaC:** В отличие от скриптов автоматизации (например, Bash, где ты пишешь пошаговый алгоритм создания ресурсов), в Terraform ты описываешь **целевое состояние** (Target State). Ты говоришь: *«Мне нужна сеть VPC и 3 виртуальные машины EC2»*. Если инфраструктура пустая, Terraform создаст их. Если виртуальные машины уже есть, но их 2, Terraform досоздаст всего одну.
* **Idempotency (Идемпотентность):** Повторные запуски одного и того же кода Terraform с одинаковыми входными данными всегда приводят к одному и тому же результату, предотвращая дублирование ресурсов.

---

### 2. Управление состоянием (State Management)

Ключевое отличие инструментов инициализации (Terraform) от инструментов управления конфигурацией (Ansible) — наличие **State файла (`terraform.tfstate`)**. Это файл в формате JSON, который хранит точную карту соответствия между твоим HCL-кодом и реальными ресурсами в облаке.

#### Зачем нужен State-файл?
1. **Mapping:** Код оперирует именами вроде `aws_instance.web`. Облако выдает ресурсу ID вроде `i-0abcdef12345`. State связывает их воедино.
2. **Performance:** Чтобы понять, что изменилось, Terraform не нужно делать тысячи запросов к API AWS при каждой команде — он берет кэш из State (хотя перед деплоем актуализирует его через операцию `refresh`).
3. **Управление зависимостями:** State знает граф зависимостей ресурсов, чтобы корректно и безопасно их удалять или изменять.

#### Локальный vs Удаленный State (Remote State & Locking)
* **Проблема локального State:** Если ты хранишь `terraform.tfstate` на своем ноутбуке, твои коллеги не смогут работать с этой инфраструктурой. Если вы запустите код одновременно, вы создадите дубли или затрете конфигурацию друг друга. Заливать State в Git **категорически запрещено**, так как он содержит секреты (пароли от БД, приватные ключи) в открытом виде.
* **Enterprise-решение:** Использование **Remote Backends** (например, AWS S3, Google Cloud Storage). Файл состояния хранится в защищенном облачном бакете. 
* **State Locking (Блокировка состояния):** Чтобы предотвратить одновременное изменение инфраструктуры двумя инженерами или параллельными CI/CD пайплайнами, используется механизм блокировок. В случае с AWS S3, Terraform использует таблицу **Amazon DynamoDB**. При запуске команды таблица переводится в статус «заблокировано», и второй процесс завершится ошибкой, пока первый не закончит работу.

### 3. Провайдеры (Providers)

Terraform ничего не знает про устройство API конкретных облаков. Всю работу по трансляции декларативного HCL-кода в императивные REST API запросы выполняют **Провайдеры**.

Провайдер — это исполняемый плагин (бинарник на Go), который скачивается автоматически при инициализации проекта (`terraform init`) из публичного реестра (Terraform Registry).

#### Жизненный цикл изменений (Core Workflow)

1. **`terraform init`:** Скачивает необходимые плагины провайдеров и инициализирует backend для хранения State.
2. **`terraform plan`:** Сравнивает текущий код, локальный State и реальное состояние ресурсов в облаке. Генерирует сухой остаток: что будет создано (`+`), изменено (`~`) или удалено (`-`). На этом этапе API облака не вызывается на изменение.
3. **`terraform apply`:** Повторно проверяет план, запрашивает подтверждение и начинает отправлять реальные запросы в API провайдера для изменения инфраструктуры. По окончании записывает изменения в State.
4. **`terraform destroy`:** Полностью уничтожает все ресурсы, описанные в текущем проекте, строго соблюдая обратный порядок графа зависимостей.

### 4. Повторно используемые модули (Reusable Modules)

**Модуль** в Terraform — это контейнер для нескольких ресурсов, которые используются вместе. По сути, это аналог функций в программировании, позволяющий избежать копипасты (DRY — Don't Repeat Yourself).

#### Идеальная структура переиспользуемого модуля:
Любой кастомный модуль (например, для разворачивания стандартного инстанса PostgreSQL под Prisma бэкенд) должен состоять минимум из трех файлов:

```text
postgresql-module/
├── main.tf      # Основная декларация ресурсов (инстанс, диски, подсети)
├── variables.tf # Входные аргументы модуля (размер диска, тип инстанса, env)
└── outputs.tf   # Выходные параметры (IP-адрес, endpoint базы данных для других модулей)
```

#### Пример использования модуля в основном коде проекта:
```hcl
module "production_db" {
  source = "./modules/postgresql-module" # Путь к модулю (локальный или Git-репозиторий)

  # Передача входных переменных (variables.tf)
  instance_type  = "db.t3.medium"
  allocated_storage = 50
  environment    = "production"
}

# Использование выходного параметра модуля в другом ресурсе
resource "aws_route53_record" "db_dns" {
  zone_id = var.dns_zone_id
  name    = "db.agentic-news.internal"
  type    = "CNAME"
  ttl     = 300
  records = [module.production_db.db_endpoint] # Взяли output из модуля
}
```
